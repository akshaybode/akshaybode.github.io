1. The Ways to Access AWS

There are a few ways to access resources you've created (S3 or anything else). We access S3 most often, so it's the running example.

WayWhat it isNotes
AWS Portal (UI)The web console you click around in.Most common for ad-hoc work; great UI.
AWS CloudShellA terminal built into the portal (the shell icon in the top bar).Run CLI commands in the browser — create/delete resources, upload files. Still "in the portal".
Local CLIInstall the AWS CLI on your own machine.Quick one-liners; uses REST APIs under the hood.
SDK (e.g. boto3) / REST APICall AWS from real code in your editor.Closest to real development — the DE's preferred way.
Interview-tricky point: The fundamental unit is the REST API. The SDK, the local CLI, and the portal are all just wrappers built on top of REST APIs — and this is true for every cloud provider, not just AWS.

2. The Security Setup (recap of IAM in action)

Say Anna wants to access an S3 bucket from her local VS Code instead of the UI. S3 is secured, so two things are required:

  • Permission (policy): Anna must have an S3 policy, e.g. AmazonS3FullAccess (her "admission to the university").
  • Identity proof (access keys): Anna must prove she's Anna — her "ID card" shown to the security guard. That ID card is an access key.

Creating access keys

IAM → Users → Anna → Create access key (use case: local code). You get two values:

  • Access Key ID — the public identifier.
  • Secret Access Key — the secret value (shown only once — save it now).
Never hardcode or expose secrets. Treat the secret access key like a password. Store it in a .env file (below), never paste it into shared code.

3. Accessing S3 from Python (boto3)

Step 1 — Store secrets in a .env file

# .env  (never commit this file)
AWS_ACCESS_KEY_ID=your_access_key_here
AWS_SECRET_ACCESS_KEY=your_secret_value_here

Step 2 — Create a virtual environment & install packages

python -m venv .venv
.venv\Scripts\activate        # Windows
pip install boto3 python-dotenv

(boto3 is the AWS SDK for Python; python-dotenv loads the .env file.)

Step 3 — Create a client and use it

import os
import boto3
from dotenv import load_dotenv

load_dotenv()  # loads AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY from .env

s3_client = boto3.client(
    "s3",
    region_name="ca-central-1",  # your bucket's region
    aws_access_key_id=os.environ.get("AWS_ACCESS_KEY_ID"),
    aws_secret_access_key=os.environ.get("AWS_SECRET_ACCESS_KEY"),
)

# List all buckets
buckets = s3_client.list_buckets()
for b in buckets["Buckets"]:
    print(b["Name"], b["CreationDate"])

Running this prints every bucket name and creation date — proof your local machine (with no direct AWS link) is talking to AWS through the SDK.

More operations

The client exposes tons of functions: upload_file / put_object, copy, list_objects, delete_bucket, and more.

# Delete an (empty) bucket — note expected owner = your 12-digit account ID
s3_client.delete_bucket(
    Bucket="demo-bucket-name",
    ExpectedBucketOwner="123456789012",
)
Got "Access Denied" deleting a bucket? Use your real 12-digit account ID for ExpectedBucketOwner — an alias won't work (many accounts can share an alias). Also, a bucket must be empty before deletion.
Don't memorize the code. All of it is in the official boto3 documentation (AWS SDK for Python), organized per service (search "S3", "Glue", etc.) with every function and parameter described. You can also let AI generate it — the value is knowing the concepts (client, credentials, region).

4. Why this matters for interviews & the job

  • Everyone can click around S3 in the portal. What makes you stand out is talking about boto3 clients, creating them programmatically, and automating tasks in code.
  • Real day-to-day work needs scripts — e.g. a Python script that deletes stale data on a schedule. You'd often run such scripts inside a Lambda function (next part), where you don't need access keys because the function uses an attached IAM role instead.
Recap of Part 5: Four ways to access AWS — Portal, CloudShell, local CLI, and SDK — but all are wrappers over REST APIs. To use the SDK you need a policy (permission) plus access keys (identity). In Python: store keys in .env, load with python-dotenv, create a boto3.client("s3", region, keys), and call functions like list_buckets(). Inside AWS (e.g. Lambda) you skip keys and rely on an IAM role.
