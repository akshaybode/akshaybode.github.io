1. What is AWS Glue?

As the name suggests, Glue binds multiple components together. A real data-engineering solution has many isolated pieces — extract from source, transform with Spark, load to a warehouse, then report. Glue manages the order and dependencies between those steps. That's why it's called an ETL solution.

Glue orchestrates the ETL stages (in the right order)
Extractfrom source
→
TransformSpark / logic
→
Loadwarehouse / lake
→
ReportBI / dashboards
Best part: Glue is a low/no-code solution. Even without writing Spark, you can build an entire pipeline visually (and it generates the code for you).

Glue's two components

  • Data Catalog — where you register tables (metadata: table/column names, data types) so the data can be queried with SQL.
  • Visual ETL — build the actual pipeline using low/no-code (or notebooks / script editor).
Glue isn't "created" like a bucket. It's a layer/component you use — you create objects (databases, tables, jobs) inside it.

2. Data Catalog & Crawlers

Why register tables at all? Your data sits in S3 as CSV/Parquet files. To let anyone query it with SQL, you must register it as tables in the catalog. Two ways:

OptionHowVerdict
Crawler (automatic)Point a crawler at an S3 path; it walks every file/folder/subfolder, infers the schema, and registers tables automatically.Fast & preferred — especially for 50–100 column files.
Manual tableDefine the schema yourself (column name + type, or JSON).Error-prone for wide tables; fine for small/known schemas.

Hands-on

  • Glue → Databases → create one (e.g. demo_db).
  • Glue → Crawlers → Create crawler → name it → add S3 data source (e.g. the raw/ folder) → "crawl all subfolders".
  • IAM role: the crawler needs to read S3 — create/assign a role like glue-access-s3 (Glue + S3 access).
  • Target database = demo_db; schedule = on demand (running it costs, since it crawls everything).
  • Run crawler → in seconds it registers a raw table with the exact inferred schema. You can edit column data types (e.g. quantity → integer) afterward.

3. Visual ETL — Build a Pipeline (low/no-code)

Glue → Visual ETL → create job. The canvas uses nodes, where each node is one activity (extract / transform / load). Example pipeline: read CSV from raw/, drop a column, write to processed/ in Delta format (a lakehouse).

Demo pipeline (each box = a node)
Source: S3raw/ CSV
→
TransformSelect fields (drop order_date)
→
Target: S3processed/ Delta
  • Source node (Amazon S3): set the location (raw/), data format (CSV), and enable "Infer schema" so column names flow to later steps.
  • IAM role: give the job S3 access (e.g. glue-access-s3). Start a session to use Data preview (runs Spark under the hood to show live output).
  • Transform node (Select fields): pick the columns to keep (e.g. drop order_date).
  • Target node (Amazon S3): choose plain "Amazon S3" (just dumps data) vs "S3 + Data Catalog" (dumps and registers). Set format = Delta Lake, target = processed/. Other targets available: Snowflake, BigQuery, etc.
"Infer schema" matters. Without it, transformations can fail because the columns aren't known. The demo intentionally failed first, then succeeded after enabling Infer schema.

Run, script & monitor

  • Run the job; check Job runs to monitor success/failure (takes ~2 min).
  • Script tab: the visual flow auto-generates the equivalent Glue/PySpark code — you can switch to script-only if you prefer code.
  • Version control: connect Git (GitHub/GitLab/Bitbucket/Azure DevOps). Git is the technology; GitHub is a provider of it.
  • After success, processed/ contains the Delta data (with a _delta_log).
Why low/no-code is trending: like a food chain (McDonald's) using templates vs a single chef's custom dish — templated pipelines are easier to manage, scale, and hand over when vendors leave. Anyone can follow the template.

4. AWS Athena — Serverless SQL

Athena is an interactive, serverless SQL query service over S3 (like Azure's serverless SQL pool). You've stored, transformed, and cataloged data — now you query it. Athena is an abstraction layer: you don't query S3 directly (it's unstructured/semi-structured), you query Athena, and it runs SQL against the files.

Athena = SQL abstraction layer over S3
👤 Userwrites SQL
→
🔍 Athenaquery engine
→
📁 S3 filesCSV / Parquet / Delta / JSON

Glue and Athena share the same Data Catalog

This is the key insight: Glue and Athena are both connected to the same Data Catalog. Register a table via Glue's crawler, and Athena can immediately query it — and vice versa.

-- Query the table the crawler registered
SELECT * FROM demo_db.raw;

Creating tables in Athena (external tables)

Athena → Create → "table from S3". Pick the database (demo_db), the S3 location, and format:

  • CSV: you must define columns (use bulk add columns, e.g. order_id int, customer string, ...).
  • Delta/Parquet: Athena auto-generates the CREATE EXTERNAL TABLE for you.

Athena creates an external table — just a table definition / metadata pointing at the real data in S3. The actual data stays in the data lake; the table is only a schema on top of it. You can save queries (e.g. DDL_demo) and reuse them.

Crawler from Athena? Athena offers "create table using crawler", but clicking it just sends you to Glue — because crawlers live inside Glue. Preference: use Athena to create tables for Delta/lakehouse data, and Crawler for other formats.

5. The Complete End-to-End Pipeline

Now glue everything together into one real-world mini use case: a user uploads a CSV to S3, which triggers a Lambda that starts the Glue ETL pipeline, whose output is then queryable in Athena.

The Lambda that triggers the Glue job

Create a new function (e.g. trigger-lambda) and give its role three permissions: AWSLambdaExecute (basic), S3 access (to listen for uploads), and Glue access (to start the job). The code creates a Glue client and calls start_job_run:

import boto3

def lambda_handler(event, context):
    glue = boto3.client("glue")   # uses the function's IAM role
    glue.start_job_run(JobName="demo-pipeline")
    return {"statusCode": 200, "body": "pipeline triggered"}
  • Deploy → Test: response succeeds and the Glue job starts (check Glue → the job → Runs).
  • To fully automate, add an S3 trigger (PUT) just like in Part 6 — now any upload kicks off the whole pipeline.
  • Find the function name (e.g. start_job_run) in the boto3 docs under the Glue service — don't memorize it.
Recap of Part 7 & the whole masterclass: Glue is the ETL "glue" with a Data Catalog (tables, populated fast by Crawlers) and a Visual ETL builder (low/no-code, auto-generates Spark). Athena runs serverless SQL as an abstraction layer over S3 via external tables, sharing the same catalog as Glue. Tie it together: an S3 upload triggers a Lambda that starts a Glue job, and the transformed data is queried in Athena — a complete, real-world data pipeline. 🎉
