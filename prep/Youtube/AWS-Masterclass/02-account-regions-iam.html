1. Creating a Free AWS Account

AWS gives new learners a generous free tier for 6 months (Azure gives only 30 days). You also start with ~$100 in credits, plus more credits as you complete activities (building services for learning). Some services — like IAM — are always free.

  • Go to the AWS site → Create account → Sign up.
  • Provide your root user email (creates the all-powerful root user).
  • Provide an account name and card details — this stops people creating infinite free accounts.
  • Sign in to the console with root user email, password, and MFA (e.g. an authenticator app).
Best practice: Don't keep logging in as the root user. Create an IAM user for yourself (covered below) and use that day-to-day.

2. Console Walkthrough

  • Top-right corner: your account name + 12-digit Account ID (note it down — you'll need it).
  • Console Home: recently visited apps, applications, cost & usage, and free-tier days remaining.
  • Menu (☰) → All services: the full 200+ list — compute, containers, storage, databases, migration, networking, ML, IoT, blockchain…
  • Search bar: the fastest way to open a service. Type EC2, S3, Athena, etc. Forgot a name? Google it — you'll memorize them after a few uses.
Region selector: Notice the region dropdown in the top bar (e.g. CA Central 1). It controls where your resources are deployed — which leads to the next topic.

3. Regions & Availability Zones

What is a Region?

A Region is a specific geographic area in the world (Mumbai, Bangalore, Dubai, Canada Central, US-East…). When you create a resource (say a virtual machine), AWS asks where to deploy it. Key rules:

  • Deploy close to your users → lower latency, lower egress cost, faster transactions.
  • You can still use a far region — it just won't be as fast.
  • Each region is isolated — it has its own zones and depends on no other region.

What is an Availability Zone (AZ)?

Within each region there are multiple Availability Zones. Each AZ = one (or more) physical data center (where your real resources live), wrapped in an isolated boundary.

Why multiple AZs? Redundancy.

When you ask AWS to create a resource, it doesn't make just one copy — it creates three copies across three AZs. You connect to the region; under the hood it routes to AZ 1 by default. If AZ 1 goes down (power outage, disaster), AZ 2 takes over, then AZ 3.

Failover chain (automatic)
AZ 1default
→ if down
AZ 2backup
→ if down
AZ 3backup

Multi-Region Redundancy & Local Zones

  • Multi-region redundancy: if an entire region fails (rare, but nature happens), you can opt in to deploy resources across multiple regions (e.g. Bangalore + Canada Central).
  • Local Zones: an extension of a region placed closer to a dense population center (e.g. inside US-West-2, an extension in Los Angeles) to give single-digit-millisecond latency.

4. IAM — Identity & Access Management

IAM was added relatively late but is now non-negotiable, especially for data engineers who wire many services together. Search IAM in the console to open its dashboard.

The problem IAM solves: in an organization you create services for many teams (web dev, ML, data engineering). Should everyone access everything? No — you must restrict access. That's identity (who you are) + access management (what you can do). IAM has 4 core concepts:

5. Users

The first step in IAM: register people as users so they can access your AWS account. Example: a new joiner "Anna". In IAM → Users → Create user:

  • Set username (Anna), enable console access, set/auto-generate a password.
  • Anna signs in using the Account ID (or alias) + username + password, then resets the password.
  • Important: a brand-new user has no permissions → Anna sees Access Denied everywhere until you attach a policy.

6. Policies (Permissions)

A policy is just a fancy word for permissions. To let Anna do things, attach a policy. AWS ships 1,400+ pre-built policies so you rarely write your own — but you can create custom policies when you need very fine-grained control.

Example: attach AdministratorAccess (the most privileged policy). Its rule says Allow every action on every resource:

Attaching a policy unlocks access
👤 Annano access
+ policy →
AdministratorAccessAllow all actions on all resources
→
✅ Anna can do everything

In IAM → Users → Anna → Add permissions → Attach policies directly → pick the policy → Add permissions. Refresh Anna's account and she now sees everything.

7. User Groups

In real life we use groups 99% of the time instead of attaching policies to individuals. As the team grows, attaching the same policies to each person one-by-one is painful and repetitive.

Instead, create a group, attach policies once to the group, then just drop users into it — they inherit all the group's permissions automatically.

Define permissions once per group, then add people
Analytics GroupS3, Athena policies
← add users
👥 Analysts
Data Science GroupS3, SageMaker policies
← add users
👥 Data Scientists

Create via IAM → User groups → name it (e.g. analytics-group) → attach a policy (e.g. AmazonS3FullAccess) → create → then add users into it.

DE relevance: Data engineers are the upstream for analytics & data-science teams, so they typically manage these permissions. With AI's rise, senior DEs often absorb the old "cloud admin" duties — it's a quick group-management task.

8. Roles (the DE favorite)

Users/groups answer "how can a person access services?". Roles answer "how can a service access another service?". This is what data engineers manage ~90% of the time, because DEs constantly move data between services.

Example: AWS Glue wants to read/write S3. S3 is secured, so it asks "who are you? do you have permission?" Without a role, Glue's request takes a U-turn — denied.

No role = denied · With role = allowed
🠻 Glueno role
↩️ U-turn
🔒 S3access denied
🠻 Glue+ role: "any Glue can access any S3"
→
✅ S3access allowed

Creating a role

IAM → Roles → Create role → trusted entity = AWS service → choose the service (e.g. Glue) → attach permission (e.g. AmazonS3FullAccess) → name it (e.g. glue-access-s3) → create. Later, when you create a Glue instance that talks to S3, you attach this role to it. The same idea applies to Lambda, etc.

Recap of Part 2: Free tier gives 6 months + ~$100 credits. Regions are geographic areas; each contains isolated AZs (data centers) and AWS keeps 3 redundant copies for failover. IAM is the security core: Users (identities) get Policies (permissions), managed at scale via Groups, while Roles grant service-to-service access (Glue→S3, Lambda→S3).
