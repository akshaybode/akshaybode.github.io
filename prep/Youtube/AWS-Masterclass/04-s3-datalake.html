1. From Block Storage to Blob Storage

Recall block storage (Part 3): your data is split into blocks, and you need an OS / VM to reassemble and read it. The pain point: to access stored data you must log into a virtual machine.

What if you want to dump data and read it back without any VM or OS? That's what blob storage solves. BLOB = Binary Large Object.

How blob storage works

Under the hood it still uses the block-storage idea — your file is split into chunks. But instead of an OS reassembling them, the chunks live on distributed storage nodes, and you access them through APIs — you request objects, you don't "open files".

2. What is a Data Lake?

A data lake is just enhanced blob storage. Plain blob storage dumps files with no structure. For big-data analytics you also want:

  • Hierarchical folders — e.g. a sales/ folder, a categories/ folder, to keep data organized.
  • ACLs (Access Control Lists) — fine-grained permission control.

So: data lake = blob storage under the hood + folder hierarchy + finer access control.

AWS vs Azure naming: In Azure you get two services — Blob Storage and ADLS Gen2 (data lake). In AWS there's just one: S3, which is the enhanced-blob/data-lake solution.

3. S3 — Simple Storage Service

S3 is AWS's data-lake / object-storage service — the one DEs use almost weekly. Search S3 in the console. Bucket types you'll see: general purpose, directory, table, and vector buckets (newer, for AI engineering — storing embeddings/vectors).

Creating a bucket

  • Bucket name must be globally unique (across all of AWS).
  • ACLs: usually left disabled.
  • Block all public access: keep this ON — since data is reachable via API, you don't want anyone calling it. This denies all public access.
  • Bucket versioning: enable it (see below).
  • Create the bucket → open it → use Create folder (e.g. raw/) and Upload files.
The "Create folder" button itself is proof of the data-lake behavior — plain blob storage wouldn't offer hierarchical folders.

Store any form of data

A big advantage of the data lake: it stores any data — structured CSV, Parquet/Delta, JSON, text, images, video. Similar feel to Google Drive / OneDrive, but for analytics.

4. Versioning

With versioning enabled, uploading a file with the same name doesn't overwrite it — S3 keeps a new version and serves the latest by default. Click Show versions to see all versions (each with its own timestamp); you can download or restore any older version.

Why it matters: DEs often accidentally overwrite files (e.g. a parameters/config file). Versioning saves hours of recovery work — you just restore the previous version instead of reconstructing it from pipeline logs.

5. S3 Storage Classes

Not all data is accessed equally. Before storing, ask: how frequently will this be read? Pick a storage class that matches the access pattern so you don't overpay. (Similar to Azure hot/cool/cold tiers.) The default is S3 Standard.

Storage classBest for
S3 StandardFrequently accessed data (more than once a month), millisecond access. Default. ~99.999999999% durability, ≥3 AZs.
S3 Standard-IA (Infrequent Access)Long-lived but infrequently accessed (about once a month), still millisecond access.
S3 Intelligent-TieringUnknown / changing / unpredictable access — AWS auto-moves data between tiers to optimize cost.
S3 One Zone-IAInfrequent access, stored in a single AZ (cheaper, less redundancy).
S3 Express One ZoneSingle-digit-millisecond, high-performance, single zone.
Glacier Instant RetrievalArchive that you occasionally need instantly.
Glacier Flexible RetrievalLong-lived archive, retrieval in minutes/hours.
Glacier Deep Archive"Store it and forget it" — cheapest, slowest retrieval.
Don't memorize — refer. These classes change over time. As a senior DE, check the official docs when making a storage-class decision. The point is matching cost to access pattern (hot → standard, archive → glacier).
Recap of Part 4: Blob storage (BLOB = Binary Large Object) splits files into distributed chunks accessed via APIs — no VM needed (unlike block storage/EBS). A data lake is enhanced blob storage with folders + ACLs, and in AWS that's S3. Create globally-unique buckets, keep public access blocked, enable versioning to avoid lost overwrites, and pick a storage class (Standard → IA → Glacier) to match how often data is read.
