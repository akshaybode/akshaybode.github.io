Virtualization, EC2 & EBS

The most important infrastructure concept in cloud: how one giant server becomes many isolated virtual machines, how to launch them with EC2, and how disks (EBS) attach to them.

1. Virtualization

A server is just a machine like your laptop — only far more powerful (much more RAM, storage, processors). Imagine combining the RAM/CPU/storage of every computer on your street into one machine.

Virtualization = slicing that one big, powerful server into many small, isolated virtual machines (VMs).

One powerful server → many isolated VMs
🧮 One big physical server (shared RAM / CPU / storage)
VM 1Windows, 8GB
VM 2Ubuntu, 16GB
VM 3Mac, 32GB

Two key properties of each VM

  • Own operating system — each VM runs its own OS (Windows/Linux/Mac); they don't share the OS. This makes each VM isolated from the others.
  • Shared resources, allocated quota — RAM/CPU/storage all come from the shared server, but each VM gets a dedicated allocation (e.g. "give me 8GB RAM, 1TB storage"), so it feels private.

Why virtualize?

Before cloud, if you needed Windows + Linux + Mac machines occasionally, buying separate machines was wasteful. Instead you bought one big server and carved out VMs on demand. After cloud, you don't even buy the server — AWS owns it, and you just request "give me a Windows VM with 8GB RAM" and AWS creates it for you.

2. Hypervisor — the magic behind virtualization

A hypervisor is the software that makes virtualization possible: it creates the VMs and allocates CPU/RAM to each. No hypervisor → no virtual machines.

TypeWhat it isExamples
Type 1 (bare metal)Runs directly on the hardware.VMware ESXi, Microsoft Hyper-V
Type 2 (hosted)Runs as software on top of a host OS.VirtualBox, VMware Workstation
For cloud: the provider manages the hypervisor for you. You don't worry about which one AWS uses — just know that a hypervisor is the engine creating and powering your VMs.

3. EC2 — Elastic Compute Cloud

EC2 (Elastic Compute Cloud) is simply a virtual machine you rent from AWS. It's the flagship product of every cloud provider — thousands of instance types (combinations of processor, storage, networking, OS, even GPUs).

Launching an instance

Search EC2 → Launch instance:

  • Name the server.
  • AMI / OS: pick from free-tier images — Amazon Linux, Ubuntu, Windows Server, macOS, etc. (Amazon Linux is very popular in AWS; it's a Linux distro like Ubuntu.)
  • Instance type (CPU/RAM size).
  • Key pair: create one (downloads a .pem file — keep it safe; you'll need it to decrypt the Windows password / SSH in).
  • Network: allow HTTP/HTTPS traffic (to hit endpoints) and SSH from anywhere (to connect).
  • Storage: 8 GB is enough for learning. Then Launch instance — ready in seconds.

Linux vs Ubuntu (quick aside)

Linux is the kernel (the engine). Combine that kernel with utilities, a terminal and tooling and you get a distribution like Ubuntu or Amazon Linux. You interact with the kernel through the distro.

Connecting to the instance

OSHow to connect
Linux (Amazon Linux/Ubuntu)Use EC2 Instance Connect (browser terminal) or an SSH client. No UI — pure terminal (pwd, cd, ls). Fast, because no resources wasted on a GUI.
WindowsUse RDP (Remote Desktop). Download the .rdp file, get the password (decrypt it with your .pem key pair), connect → full desktop UI. Slower, because resources go to the GUI.
Always stop instances you're not using! A running instance keeps billing. Instances → select → Instance state → Stop. Don't waste your free credits.
Windows vs Linux trade-off: Windows VMs need resources allocated to the UI (extra cost), which is why core developers often prefer fast, UI-less Linux machines and work via the terminal. Big orgs may still use Windows for easier management.

4. EBS — Elastic Block Storage

To understand EBS, first understand block storage.

What is block storage?

When you save a file (say document.txt, 100 MB) on your computer, the OS doesn't store it as one chunk. It breaks the file into small blocks (typically KB-sized), stores them, and keeps metadata about where each block lives. When you open the file, the OS finds all the blocks, reassembles them, and shows you one file — you never see the blocks.

Block storage relies on the OS to split & reassemble
📄 Filedocument.txt
→ OS splits
◫◫◫ Blocks+ metadata
→ OS reassembles
📄 Full fileshown to you

Key point: block storage needs an operating system / VM in the picture to manage metadata and reassemble blocks.

EBS = block storage for EC2

Elastic Block Storage (EBS) provides scalable, high-performance block-storage volumes (HDD or SSD) that attach to an EC2 instance — think of it as a hard disk for your VM. You cannot access EBS data without the VM/OS.

DE relevance: Data engineers rarely touch EBS after the VM is set up — it's a once-in-a-few-months task. The storage service DEs use constantly is S3 (next part). EBS is for VM disks; S3 is for data lakes.
Recap of Part 3: Virtualization slices one powerful server into isolated VMs (each with its own OS, shared-but-allocated resources), powered by a hypervisor. EC2 is a rented VM — launch Linux (terminal) or Windows (RDP) in seconds, and always stop it when idle. EBS is block storage (a disk) that attaches to an EC2 instance and depends on the OS.
