1. What is AWS Lambda?

AWS Lambda lets you run code without provisioning or managing servers — that's the entire story. Write a function (Python, Node.js, Java, Go, …), upload it (as a file or container image), and Lambda automatically allocates compute and runs it on each incoming request/event, at any scale.

(Coming from Azure? It's like Azure Functions.)

Why not just use EC2?

You could run your Python on an EC2 VM. But if the code only runs a few times a day, the VM sits idle the rest of the time — wasted money — and you must scale and manage it yourself.

EC2 (always on) vs Lambda (on demand)
EC2idle most of the day · you manage it
vs
Lambdaruns on demand · pay per run · auto-scales

Lambda is a serverless compute/interpreter that:

  • Charges you only for the time it runs (e.g. 3,300 ms & 128 MB used).
  • Auto-scales up and down with no idle waste.
  • Can be triggered on demand by events.
  • Needs no OS/interpreter upgrades or installs — just submit code and run.
Not a full EC2 replacement. Lambda is for small, on-demand functions — not heavy, long-running, critical workloads. Use the right tool for the job.

2. Creating a Lambda Function

Search Lambda → Create function → Author from scratch:

  • Name it (e.g. lambda-demo).
  • Runtime: pick a stable Python (e.g. Python 3.12 — avoid brand-new releases with bugs), arch x86_64.
  • Execution role: Create a new role with basic Lambda permissions — this auto-creates a role that lets the function execute (basic permission). You can add more permissions later.

You get an embedded code editor (VS Code-like, with AI code completion) right in the console.

The handler: event and context

def lambda_handler(event, context):
    print("event is", event)
    return {
        "statusCode": 200,
        "body": "Hello from Lambda!"
    }
  • event (important): the input coming from whatever invoked the function (a trigger, or a test event). Use it to drive your logic.
  • context (less important): metadata about the invocation.

Deploy & Test

There's no "run" button — click Deploy to save your code (it's a live connection, so deploy after every change). Then Test: create a test event (a JSON input) and run it. You'll see the response, e.g. statusCode 200 and your body. The print(event) output appears in the logs.

3. Monitoring with CloudWatch

  • Monitor tab: shows all invocations of the function over time (in real apps this is busy — functions get called by many APIs, sometimes asynchronously).
  • CloudWatch logs: click through to see each invocation, its request ID, the event passed in, your print output, memory used, etc.
CloudWatch is AWS's monitoring service (like Azure Monitor). Every Lambda run is logged there — your first stop when debugging.

4. Triggers

Instead of testing manually, attach a trigger so the function runs automatically on an event. Add trigger shows a long list: API Gateway, IoT, S3, and many more.

Setting up an S3 trigger

  • Add trigger → source = S3 → pick the bucket.
  • Event type: PUT (fires when a file is added/created).
  • Prefix: optionally limit to a folder (e.g. trigger/).
Beware recursive loops! If the function writes back into the same bucket that triggers it, it can re-trigger itself forever (AWS will email you a recursion warning). Read from one location and write to another, or delete the source file.

5. Execution Role (custom, reusable)

For the function to touch S3 it needs permission. The auto-created role only has basic execute rights, so create a custom role you can reuse across many functions.

IAM → Create role → service = Lambda → attach AWSLambdaExecute (basic + S3 get/put) and AmazonS3FullAccess → name it lambda-access-s3. Then assign it to the function via Configuration → Permissions → Edit → Existing role.

Inside AWS, no access keys needed. Because the function runs within AWS, it uses its attached role for identity — so your boto3 client doesn't pass any keys/secrets.

6. Example — Move a File on Upload

A function that copies a file from one folder to another in the same bucket (then deletes the original to avoid a recursive loop):

import boto3

def lambda_handler(event, context):
    s3 = boto3.client("s3")   # no keys needed: uses the function's IAM role
    bucket = "my-bucket-26"
    source_key = "trigger/data1.csv"
    dest_key = "destination/data1.csv"

    # Copy then delete the original (prevents recursive trigger loop)
    s3.copy_object(
        Bucket=bucket,
        CopySource={"Bucket": bucket, "Key": source_key},
        Key=dest_key,
    )
    s3.delete_object(Bucket=bucket, Key=source_key)

    return {"statusCode": 200, "body": "file moved successfully"}

Deploy, then upload data1.csv into the trigger/ folder. Check Monitor → CloudWatch logs to confirm it ran, then verify the file now sits in destination/ and is gone from trigger/.

Common gotchas from the demo: the handler must be named correctly (default lambda_handler, not a renamed function), and the bucket name in code must exactly match the real bucket — otherwise you'll see "object not found".
Recap of Part 6: Lambda runs code serverless — pay per run, auto-scale, no servers to manage. The handler takes event (input) and context (metadata); Deploy to save, Test to run, and watch CloudWatch logs. Attach triggers (e.g. S3 PUT) to run automatically, give the function a custom IAM role for permissions, and (inside AWS) skip access keys. Watch out for recursive loops.
