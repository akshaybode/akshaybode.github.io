Fundamentals & the Data Engineering Lifecycle

What data engineering really is, where it sits, the roles around it, and the lifecycle every data system follows.

1. Why Data Engineering Exists

Businesses run on the internet — Amazon (e-commerce), Netflix (streaming), Zomato (food delivery). Each has goals: understand the customer, increase profit, improve processes. Humans take many decisions on assumptions and experience, but experts still miss things they can't see.

The most reliable way to be right most of the time is to look at what the data says. This is why Data Engineering, Data Science, Machine Learning, and AI exist — to turn raw activity into factual understanding that drives the business.

The one rule to remember: Anything you build must create value for the business — save cost, improve a process, or understand the customer. If it doesn't impact the final business outcome, it's useless no matter how advanced it is.

2. Where Data Engineering Fits

Every click, like, comment, or purchase on an app generates data. That data lands in a DBMS (Database Management System). From there, the entire data pipeline begins.

The end-to-end internet data system
📱 ApplicationUser generates data
→
🗄️ DBMSData stored (CRUD)
→
🔧 Data EngineeringThe "plumber"
→
🤖 DS / ML / 📊 DashboardsDecisions & products

Data also comes from beyond apps — sensors (a truck moving A→B→C), stock-market feeds, IoT devices, and more. Data Engineering is like a plumber: it connects one system to another, transforming data on the way. Without data, there is no DE, DS, or ML.

3. The Roles Around Data

Titles blur in small companies (one person does everything) but separate in large ones. Focus on the core responsibility, not the title.

Software Engineer

Builds & deploys the web/mobile app — writes code, ships features.

DBA (Database Admin)

Designs the database — tables, columns, schema. Sometimes done by SWEs/DEs in smaller firms.

Data Engineer

Writes ETL pipelines, builds databases/warehouses, processes big data (Spark, Kafka), integrates sources, ensures quality & governance.

Data Analyst

Explains the past — e.g., revenue this year vs last 5 years. Finds patterns.

Data Scientist

Predicts the future — e.g., next 6 months of product sales from past patterns.

ML Engineer

Automates & deploys models in production — recommendation engines (Amazon, Instagram Reels).

DS vs ML: the line is thin. In small orgs the same person does both; in large orgs they specialize. Other names (DevOps, DataOps) are just fancy labels for overlapping work.

Data Engineer's job in one line: take data from a source (RDBMS, API, sensors), transform it, and hand structured data to DS/ML so they can build dashboards and models.

4. The Data Engineering Lifecycle

The core mental model from the book Fundamentals of Data Engineering. You can build the whole pipeline with even the "shittiest" tool once you understand these stages — tools take ~30 minutes to learn, fundamentals are forever.

5 stages + 6 undercurrents
1. GenerationAPIs, RDBMS, sensors, logs, analytics
→
2. IngestionAuto-fetch new data into system
→
3. StorageOne central location
→
4. TransformationBusiness logic on raw data
→
5. ServingML, analytics, reporting
🔒 Security 📚 Data Management ⚙️ DataOps 🏛️ Data Architecture 🪢 Orchestration 💻 Software Engineering

Stage by stage

StageWhat happens
GenerationData is created across many places: APIs, RDBMS, sensors, Google Analytics, logs.
IngestionBuild a programmatic connection so that whenever new data appears (e.g., a new RDBMS row), it's automatically pulled into your system.
StorageAll ingested data is stored at a central location (lake/warehouse/object storage).
TransformationThe core of DE — apply business logic to convert raw data → structured data.
ServingDeliver clean data to ML engineers, analysts, and dashboard tools to create value.

5. Transformation — the Heart of DE

Transformation is a set of business logic applied between raw (ingested) data and served data.

Example: unifying date formats from two sources
API datadate = YYYY-MM-DD
2024-06-01
↘
🔧 TransformationJoin on ID +
standardize to YYYY-MM-DD
↗
RDBMS datadate = MM-DD-YYYY
06-01-2024
↓
Unified outputOne consistent date format

Transformation can be any of these:

  • Standardizing formats (dates, units, data types)
  • Removing duplicate or null values
  • Aggregating data; merging two datasets
  • Generating a new column from existing columns (concatenation)
  • Filtering rows

Done with Python, PySpark, Scala, SQL, or any tool. The goal: organize data into a proper structure so DS/ML can easily understand and use it.

6. The Undercurrents (Bottom Layer)

The lifecycle stages are the "top layer". Underneath, six undercurrents support everything you build.

UndercurrentMeaning
🔒 SecurityOnly the right, authorized people can access data and systems. Don't give everyone access.
📚 Data ManagementGovernance: easily find data, know each column's definition, track accountability (who owns it), data modeling, and data integrity (data is correct).
⚙️ DataOpsLike DevOps for data: automate deployment, monitor pipelines, observability, incident reporting & alerting.
🏛️ Data ArchitectureAnalyze trade-offs and design the right system to add business value (whole section in Part 4).
🪢 OrchestrationCoordinate & schedule jobs and manage dependencies between pipelines (e.g., Apache Airflow).
💻 Software EngineeringApply programming best practices: design patterns, scalability, testing, debugging to your ETL code.
Recap of Part 1: DE exists to create business value from data. It sits between app/DBMS and DS/ML, acting as a plumber. The lifecycle is Generation → Ingestion → Storage → Transformation → Serving, supported by 6 undercurrents. Transformation (applying business logic) is the heart of the job.
