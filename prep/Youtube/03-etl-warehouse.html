3. Why a Data Warehouse? (Normalization Problem)

Relational models are normalized — many tables, minimal duplication. To answer "which user bought which product and what did they pay?", you may need to join 4+ tables (User → Order → Product → Payment).

Relational/OLTP databases are not designed for analytical workloads. Aggregating across millions of rows means scanning row by row, then picking one column — slow and painful.

Solution: store analytical data in a data warehouse (OLAP). But you can't just dump it in — you use a method called dimensional modeling.

4. Dimensional Modeling: Facts & Dimensions

Just like data modeling structures a relational DB, dimensional modeling structures a warehouse. It has two table types:

Table typeStoresExamples
Fact table (center, usually one)Quantitative, measurable valuessales amount, quantity sold, revenue, profit, price, weight
Dimension tables (many)Descriptive attributes / categoriesproduct name, product category, user name, user city, date

The fact table sits at the center; dimensions surround it and join via common keys (e.g., product_id). To analyze product + quantity you just join 2 tables — and the OLAP engine handles it efficiently.

Two ways to arrange a dimensional model.

Fact table in the center, dimension tables directly attached — looks like a star.

❄️ Snowflake Schema

A more normalized version — dimensions have their own sub-dimension tables attached. Resembles a relational model but still has a fact table in the middle.

Snowflake schema

Product Dim
Category Sub-Dim
Date Dim
Month / Year Sub-Dim
Don't confuse: the Snowflake schema (a modeling concept) is different from Snowflake the company that offers a database-as-a-service.

6. Slowly Changing Dimensions (SCD)

Facts (price, quantity) change frequently. Dimension values (product name, user address) change rarely — but when they do, how do we handle history? That's what SCD strategies define.

TypeStrategyExample: city New York → New Jersey
SCD Type 1Overwrite — no history keptNew York is replaced by New Jersey. Old value gone forever.
SCD Type 2Full history — add a new row each changeAdd new row with an is_active flag (or version number / date ranges). Old rows kept. Move to Miami → another row.
SCD Type 3Partial history — current + previous columnsprevious_city = New York, current_city = New Jersey.
SCD Type 6Hybrid of 1 + 2 + 3Captures current city, previous city, start date, end date, and active flag together.

7. Data Marts

A data mart is a subset of a data warehouse built for one department/team's needs.

Company → Departments → Teams → Data Marts
🏢 Data WarehouseAll facts & dimensions
→
📦 Shipping Data MartOnly users + payment + product (100 of 300 cols)

Example: the shipping team only needs users, payments, products. They pick the ~100 relevant columns (out of 300) and build their own reporting tables. Each department solving its own problem helps the company move forward.

8. Data Lake vs Data Warehouse

Changing schema in a warehouse is hard (add a column → all old rows become null, must rework the pipeline). A data lake solves this: dump all raw data as-is (CSV, JSON, Parquet) into object storage (S3, Azure Blob/ADLS, Google Cloud Storage). You apply structure only when you read it (schema on read).

Schema on read
Raw filesCSV, JSON, Parquet
→
Data Lake (S3)Dump as-is, central repo
→
Read on demandAthena / Spark / Python — pick only what you need
Data WarehouseData Lake
DataStructured, predefined, smaller, relationalRaw, unstructured, large, undefined
UsersBusiness analystsData analysts & data scientists
Use caseBatch processing, BI reportingStream processing, ML, real-time analysis
SchemaSchema on writeSchema on read
Neither is "bad": warehouses give structured-data flexibility for analysis; lakes give you access to any data anytime. Both have a place in a real architecture.
