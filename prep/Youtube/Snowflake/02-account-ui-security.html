1. Account Setup (Free Trial)

  • Go to signup.snowflake.com and start a 30-day free trial with $400 of credits.
  • Pick an edition (Enterprise is recommended for the trial so all features work) and a cloud provider (AWS, Azure, or GCP) plus a region.
  • You'll get an account URL / account identifier and an email to activate the account and set your password.
  • The first user you create is automatically an ACCOUNTADMIN (the most powerful role).
Tip: the URL is your account locator — keep it; you log in through it every time. The trial credits expire after 30 days or when the $400 runs out, whichever comes first.

The modern web UI is called Snowsight. The main areas in the left navigation:

3. Users & Roles — RBAC

Snowflake security is Role-Based Access Control. The golden rule: privileges are granted to roles, and roles are granted to users (never grant directly to a user). A user can have several roles and switches between them.

▶ Privilege → Role → User (animated — grants flow downward)
🔑 PrivilegesSELECT, INSERT, CREATE…
🎭 Rolea bundle of privileges
👤 Usera person / service

The system-defined role hierarchy

Snowflake ships with built-in roles arranged in a hierarchy. A higher role inherits the privileges of roles granted to it.

Top of pyramid = most powerful
ACCOUNTADMINtop role; sees billing & everything; use sparingly
▲
SECURITYADMINmanages object grants account-wide; manages USERADMIN
▲
SYSADMIN  •  USERADMINSYSADMIN: create warehouses/DBs/objects · USERADMIN: create users & roles
▲
PUBLICdefault role every user & role has; lowest privileges
RolePurpose
ACCOUNTADMINThe most powerful role — encapsulates SYSADMIN and SECURITYADMIN. Manages account-level settings, billing, and account parameters. Grant to very few people.
SECURITYADMINCan manage any object grant globally and create/manage/monitor users and roles (inherits USERADMIN).
USERADMINDedicated to user & role management — creating users and roles, and managing grants on roles it owns.
SYSADMINCreates and manages databases, schemas, warehouses, and other objects. Best practice: all custom roles roll up to SYSADMIN.
PUBLICAutomatically granted to every user and role. Anything granted to PUBLIC is available to everyone.

Creating users, roles & granting access

-- 1) switch to the right admin role
USE ROLE USERADMIN;

-- 2) create a user
CREATE USER john
  PASSWORD = 'StrongPass123!'
  DEFAULT_ROLE = ANALYST
  MUST_CHANGE_PASSWORD = TRUE;

-- 3) create a custom role
CREATE ROLE ANALYST;

-- 4) grant privileges TO the role
GRANT USAGE ON WAREHOUSE COMPUTE_WH TO ROLE ANALYST;
GRANT USAGE ON DATABASE MY_DB    TO ROLE ANALYST;
GRANT USAGE ON SCHEMA MY_DB.PUBLIC TO ROLE ANALYST;
GRANT SELECT ON ALL TABLES IN SCHEMA MY_DB.PUBLIC TO ROLE ANALYST;

-- 5) grant the role TO the user, and roll it up to SYSADMIN
GRANT ROLE ANALYST TO USER john;
GRANT ROLE ANALYST TO ROLE SYSADMIN;
Best practices from the course: use ACCOUNTADMIN only when truly required; create custom roles for job functions (analyst, engineer, loader); always grant custom roles up to SYSADMIN so admins inherit them; follow least privilege — give each role only what it needs.

Other account-security features

Beyond RBAC, Snowflake gives admins several account-level security controls worth knowing:

FeatureWhat it does
MFAMulti-factor authentication (Duo) for user logins — strongly recommended for ACCOUNTADMIN.
Network policiesAllow/deny logins by IP range — restrict access to corporate networks/VPN.
SSO (SAML 2.0)Federated single sign-on via an identity provider (Okta, Azure AD, etc.).
SCIMAutomated user/role provisioning & de-provisioning from your IdP.
Password policiesEnforce length, complexity, rotation, and reuse rules.
Account identifierYour account is addressed as <org>-<account> (or a legacy account locator) in the login URL and connectors.

Useful inspection commands

SHOW ROLES;
SHOW USERS;
SHOW GRANTS TO ROLE ANALYST;     -- privileges this role has
SHOW GRANTS TO USER john;         -- roles this user has
SHOW GRANTS ON TABLE MY_DB.PUBLIC.CUSTOMER;
