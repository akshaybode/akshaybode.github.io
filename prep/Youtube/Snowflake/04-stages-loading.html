1. Stages — where source files live

A stage is a location that holds your data files before (or after) they're loaded into tables. There are two families: internal (inside Snowflake) and external (your own cloud bucket).

Internal stages

Snowflake-managed storage. You PUT files into them, then COPY INTO a table. There are three kinds, referenced with special prefixes:

Internal stageReferenceScope
User stage@~One per user — files only that user can see.
Table stage@%table_nameOne per table — tied to that specific table.
Named (internal) stage@stage_nameA first-class object you create & grant on. Most flexible & recommended.

External stages

Point to a bucket you own — Amazon S3, Azure Blob, or Google Cloud Storage. Snowflake reads the files in place using credentials or a storage integration (the secure, recommended way).

▶ File → Stage → Table (animated load)
📄 Source filesCSV / JSON / Parquet
📦 Stageinternal (@) or external (S3/Blob/GCS)
📋 Tablevia COPY INTO
-- named internal stage
CREATE STAGE my_int_stage;

-- external stage on S3 (using a storage integration)
CREATE STAGE my_ext_stage
  URL = 's3://my-bucket/data/'
  STORAGE_INTEGRATION = my_s3_int
  FILE_FORMAT = my_csv_format;

-- put a local file into an internal stage (run from SnowSQL/CLI)
PUT file://C:/data/customer.csv @my_int_stage;

-- inspect what's in a stage
LIST @my_int_stage;
Browser upload: in Snowsight you can also upload a file straight into a stage from the UI, then LIST it and load it — handy when you don't have the CLI installed.

Query files in a stage without loading

You can run SQL directly over staged files to preview/validate them before a COPY. Columns are referenced positionally as $1, $2, …

-- preview the first columns of staged files
SELECT $1, $2, $3
FROM @my_int_stage
  (FILE_FORMAT => 'my_csv_format')
LIMIT 10;

-- let Snowflake infer a schema from Parquet/CSV files
SELECT * FROM TABLE(
  INFER_SCHEMA(LOCATION => '@my_int_stage',
               FILE_FORMAT => 'my_csv_format'));

2. File Formats

A file format object describes the structure of your files so Snowflake can parse them: type (CSV/JSON/etc.), delimiter, header rows, compression, quoting, null handling, and more. You can define it inline in a COPY, but a reusable named object is cleaner.

CREATE FILE FORMAT my_csv_format
  TYPE = 'CSV'
  FIELD_DELIMITER = ','
  SKIP_HEADER = 1                 -- ignore the header row
  FIELD_OPTIONALLY_ENCLOSED_BY = '"'
  NULL_IF = ('', 'NULL')
  EMPTY_FIELD_AS_NULL = TRUE;

-- JSON / Parquet are just as easy
CREATE FILE FORMAT my_json_format TYPE = 'JSON';

Snowflake supports structured (CSV, TSV) and semi-structured (JSON, Avro, ORC, Parquet, XML) formats. Semi-structured data can be stored in a VARIANT column and queried with dot/bracket notation.

4. Bulk Loading with COPY INTO

The standard recipe: create the table → create a stage → put/land the files → define a file format → COPY INTO the table → validate.

-- 1) target table
CREATE TABLE customer (
  id INT, name STRING, email STRING, country STRING
);

-- 2) load all files from the stage into the table
COPY INTO customer
  FROM @my_int_stage
  FILE_FORMAT = (FORMAT_NAME = my_csv_format)
  ON_ERROR = 'CONTINUE';     -- skip bad rows instead of failing the whole load

Loading specific files & transforming on load

-- pick a specific file (or use PATTERN for a regex match)
COPY INTO customer
  FROM @my_int_stage/customer.csv
  FILE_FORMAT = (FORMAT_NAME = my_csv_format);

-- reshape columns during load with a SELECT over the stage
COPY INTO customer (id, name, email)
  FROM (
    SELECT $1, UPPER($2), LOWER($3)
    FROM @my_int_stage
  )
  FILE_FORMAT = (FORMAT_NAME = my_csv_format);

Key COPY options

OptionWhat it does
ON_ERRORABORT_STATEMENT (default), CONTINUE, SKIP_FILE, or SKIP_FILE_n — how to react to bad rows.
PATTERNRegex to select matching files in the stage.
FORCERe-load files even if already loaded (ignores load history).
PURGEDelete files from the stage after a successful load.
VALIDATION_MODEDry-run that checks files (e.g. RETURN_ERRORS) without loading.

Load history & avoiding duplicates

Snowflake remembers what it loaded. Each table keeps load metadata for ~64 days. If you run the same COPY again, already-loaded files are skipped automatically — so re-running won't duplicate rows. To intentionally reload, TRUNCATE the table (clearing the relationship) or use FORCE = TRUE.
-- validate the load worked
SELECT COUNT(*) FROM customer;

-- inspect load results / errors of the last copy
SELECT * FROM TABLE(VALIDATE(customer, JOB_ID => '_last'));

-- to reload the same file, clear the table first
TRUNCATE TABLE customer;
Bulk-load checklist: table ✓ → stage ✓ → files staged (LIST) ✓ → file format ✓ → COPY INTO ✓ → validate row count ✓.
