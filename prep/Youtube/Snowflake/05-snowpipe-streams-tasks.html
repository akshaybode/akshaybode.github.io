Snowpipe, Streams & Tasks

The automation trio: load files continuously (Snowpipe), capture what changed (Streams), and run work on a schedule (Tasks) — combine them to build a hands-off pipeline.

1. Snowpipe — Continuous Data Loading

Snowpipe automates loading of files as soon as they land in a stage, in small micro-batches, using Snowflake-managed serverless compute (no warehouse to size or pay for idle time).

A pipe is an object that wraps a COPY INTO statement. Two ways to trigger it:

  • Auto-ingest — cloud event notifications (e.g. AWS S3 → SQS/SNS) tell the pipe a new file arrived.
  • REST API — your app calls insertFiles to notify Snowpipe.
▶ Auto-ingest flow (animated — files trigger loads automatically)
📄 New file → bucket
🔔 Event notificationSQS / SNS / Event Grid
🚿 Snowpiperuns the pipe's COPY
📋 Tablerows appear automatically
CREATE PIPE customer_pipe
  AUTO_INGEST = TRUE
  AS
  COPY INTO customer
  FROM @my_ext_stage
  FILE_FORMAT = (FORMAT_NAME = my_csv_format);

-- get the notification channel (ARN) to wire up the bucket event
SHOW PIPES;
-- check the pipe's status / backlog
SELECT SYSTEM$PIPE_STATUS('customer_pipe');

2. Streams — Change Data Capture (CDC)

A stream records row-level changes (inserts, updates, deletes) made to a table since the last time the stream was read. It's how Snowflake does CDC — process only what changed instead of the whole table.

A stream doesn't store a copy of the data. It stores an offset (a point in time) plus metadata columns:

Metadata columnMeaning
METADATA$ACTIONINSERT or DELETE — the change type.
METADATA$ISUPDATETRUE if the row is part of an UPDATE (shown as a DELETE + INSERT pair).
METADATA$ROW_IDUnique id for the row, stable across changes.
 
-- create a stream on a table
CREATE STREAM customer_stream ON TABLE customer;

-- now do some DML on the base table
INSERT INTO customer VALUES (101, 'Asha', 'asha@x.com', 'IN');

-- the stream shows only the change since last read
SELECT *, METADATA$ACTION, METADATA$ISUPDATE
FROM customer_stream;
Key behaviour: reading a stream in a query does not advance it. The offset only moves forward when the stream is consumed inside a DML statement (e.g. an INSERT … SELECT FROM stream or MERGE). After consumption, the stream is "empty" until new changes occur.

Stream types: standard (tracks all DML), append-only (tracks inserts only — cheaper for append-heavy tables), and insert-only (for external tables).

3. Tasks — Scheduling & Automation

A task runs a single SQL statement (or a call to a stored procedure) on a schedule or when a predecessor task finishes. Tasks are how you orchestrate work inside Snowflake without an external scheduler.

CREATE TASK load_customer_task
  WAREHOUSE = COMPUTE_WH
  SCHEDULE = '5 MINUTE'          -- or a CRON expression
  AS
  INSERT INTO customer_history
  SELECT * FROM customer_stream;

-- tasks are created SUSPENDED; you must resume them
ALTER TASK load_customer_task RESUME;

-- CRON example: 8:00 AM every day, IST
-- SCHEDULE = 'USING CRON 0 8 * * * Asia/Kolkata'
  • Tasks can run on a warehouse you choose, or use serverless compute.
  • Chain tasks into a DAG with AFTER <parent_task> to build dependencies.
  • New tasks start suspended — remember to RESUME them.

4. Putting it together — Stream + Task pipeline

The classic pattern: Snowpipe lands data → a stream captures the new rows → a task runs on a schedule and processes only those changes (often with MERGE) into a target table.

▶ Hands-off incremental pipeline (animated, near real-time)
🚿 Snowpipeloads raw table
🔁 Streamcaptures changes (CDC)
⏰ Taskscheduled MERGE
🎯 Target tableclean / curated
-- a task that only fires when the stream has new data
CREATE TASK merge_customer_task
  WAREHOUSE = COMPUTE_WH
  SCHEDULE = '5 MINUTE'
  WHEN SYSTEM$STREAM_HAS_DATA('customer_stream')
  AS
  MERGE INTO customer_target t
  USING customer_stream s ON t.id = s.id
  WHEN MATCHED THEN UPDATE SET t.name = s.name
  WHEN NOT MATCHED THEN INSERT (id, name) VALUES (s.id, s.name);

ALTER TASK merge_customer_task RESUME;
Why this is powerful: the WHEN SYSTEM$STREAM_HAS_DATA guard means the task skips (and you don't pay for) runs when nothing changed. Snowpipe + Streams + Tasks gives you a fully automated, near-real-time, incremental ELT pipeline entirely inside Snowflake.
