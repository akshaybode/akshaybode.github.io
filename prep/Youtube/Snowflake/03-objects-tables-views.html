2. Table Types

The table type controls how long data lives, whether it's protected by Time Travel and Fail-safe, and therefore its storage cost.

TypeVisibility / lifespanTime TravelFail-safeUse for
Permanent
(default)		Persists until dropped; visible to all sessions.0–90 days*✅ 7 daysProduction data you must protect.
TransientPersists until dropped, but no Fail-safe.0–1 day❌ noneLarge, reproducible data where you want lower storage cost.
TemporaryExists only for the current session; auto-dropped on logout.0–1 day (within session)❌ noneScratch / staging data, ETL intermediates.
ExternalData stays in an external stage (e.g. S3); Snowflake only reads it.❌❌Querying data in a data lake without loading it.

* Up to 90 days requires Enterprise edition; Standard is up to 1 day.

CREATE TABLE customer (id INT, name STRING);            -- permanent
CREATE TRANSIENT TABLE stg_customer (id INT, name STRING);   -- no fail-safe
CREATE TEMPORARY TABLE tmp_customer (id INT, name STRING);   -- session only
External tables are special: the data physically remains in your cloud storage (an external stage). Snowflake stores only metadata and reads the files on demand — handy for a data lake, but slower than native tables and not protected by Time Travel/Fail-safe.
Cost tip: Fail-safe storage is billed. For huge staging/landing tables that you can always re-create from source, use transient tables to avoid paying for 7 days of Fail-safe.

3. View Types

A view is a saved SQL query that behaves like a virtual table. It stores no data (except a materialized view) — it runs its underlying query when accessed.

① Standard (non-materialized) view

  • Just a stored query. No data stored; runs every time it's queried (always fresh).
  • Use to simplify complex joins, rename/format columns, or expose a subset of data.
CREATE VIEW active_customers AS
  SELECT id, name FROM customer WHERE status = 'ACTIVE';

② Secure view

  • Like a standard view but the definition is hidden and the query optimizer won't expose internal details — protects sensitive logic and underlying data.
  • Recommended whenever a view is used for data privacy or secure data sharing.
  • Trade-off: some optimizations are skipped, so it can be a bit slower.
CREATE SECURE VIEW masked_customers AS
  SELECT id, name FROM customer;

③ Materialized view

  • Stores the precomputed results physically and Snowflake keeps them auto-refreshed as the base table changes.
  • Great for expensive aggregations queried often — fast reads at the cost of storage + maintenance compute.
  • Enterprise edition only. Has restrictions (e.g., single table, no joins, limited functions).
CREATE MATERIALIZED VIEW daily_sales AS
  SELECT sale_date, SUM(amount) AS total
  FROM sales
  GROUP BY sale_date;

Quick comparison

View typeStores data?Always fresh?Best for
StandardNoYes (recomputed)Simplifying queries; lightweight abstraction.
SecureNoYesHiding logic; data privacy & secure sharing.
MaterializedYesAuto-maintainedExpensive aggregations queried frequently (Enterprise).
Rule of thumb: default to a standard view; switch to secure when privacy/sharing matters; use a materialized view only when a heavy query is run often enough that the storage + refresh cost pays off.
