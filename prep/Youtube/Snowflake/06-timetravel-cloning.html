1. The Data Lifecycle: Active → Time Travel → Fail-safe

Every change in Snowflake keeps the old data around for a while. Data passes through three phases before it's truly gone.

What happens to old/dropped data
🟢 Activecurrent data you query normally
🔵 Time Travel0–90 days · you can query & restore
🟡 Fail-safe7 days · only Snowflake support recovers
🔴 Purgedpermanently gone

2. Time Travel

Time Travel lets you access data as it existed at a point in the past — to query historical state, recover from accidental changes, or restore dropped objects.

Retention period

  • Controlled by DATA_RETENTION_TIME_IN_DAYS (set at account / database / schema / table level).
  • Standard edition: 0 or 1 day. Enterprise+: up to 90 days for permanent objects.
  • Transient & temporary objects: max 1 day.
ALTER TABLE customer SET DATA_RETENTION_TIME_IN_DAYS = 30;

Querying the past — AT / BEFORE

-- by a timestamp
SELECT * FROM customer
  AT(TIMESTAMP => '2026-06-24 20:58:50'::timestamp);

-- by relative time: 5 minutes ago (offset in seconds)
SELECT * FROM customer AT(OFFSET => -60*5);

-- just before a specific statement ran (great for "undo this query")
SELECT * FROM customer
  BEFORE(STATEMENT => '01abc...query-id...');
Course example: insert rows, then delete some, then query the table AT a timestamp from before the delete — you'll see the rows as they were. Use SHOW TABLES to find creation time and reason about what state existed when.

Restoring dropped objects — UNDROP

Within the retention window you can instantly bring back a dropped table, schema, or database.

DROP TABLE customer;
UNDROP TABLE customer;        -- back in seconds

UNDROP SCHEMA sales_db.raw;
UNDROP DATABASE sales_db;

Restoring data into a table (recover from a bad update/delete)

-- recreate a table from its earlier state
CREATE OR REPLACE TABLE customer AS
  SELECT * FROM customer BEFORE(STATEMENT => '01abc...');

3. Fail-safe

  • A non-configurable 7-day period that begins after Time Travel ends — for permanent objects only.
  • Data here is recoverable only by Snowflake Support, not by you, and recovery may take time.
  • It's a last-resort disaster recovery measure, not a feature you query.
  • Fail-safe storage is billable — a reason to use transient tables for easily-reproducible data.
Time TravelFail-safe
Duration0–90 days (configurable)7 days (fixed)
Who recoversYou (self-service SQL)Snowflake Support only
AccessQuery & restore directlyNo direct access
Applies toAll object types (per limits)Permanent objects only

4. Zero-Copy Cloning

Cloning instantly creates a copy of a database, schema, or table without duplicating the underlying storage. The clone shares the original's micro-partitions; Snowflake only stores new data when either copy changes (copy-on-write).

Clone shares storage until something changes
📋 Original tablemicro-partitions A, B, C
⎘
🧬 Clonepoints to A, B, C (no copy)
→
✏️ Edit cloneonly the changed partition is new
CREATE TABLE customer_clone   CLONE customer;
CREATE SCHEMA raw_clone        CLONE sales_db.raw;
CREATE DATABASE sales_db_copy  CLONE sales_db;

-- combine with Time Travel: clone a table as it was in the past
CREATE TABLE customer_yesterday CLONE customer
  AT(OFFSET => -60*60*24);
  • Instant & free at creation — no storage cost until data diverges.
  • Perfect for dev/test/QA environments from production, or quick "what-if" experiments.
  • The clone is independent: changes to it don't affect the original (and vice versa).
  • You can clone at a point in time by combining with Time Travel (AT/BEFORE).
Why it's a flagship feature: spinning up a full copy of a multi-TB production database for testing is normally slow and expensive. Zero-copy cloning makes it instant and nearly free — you only pay for what you subsequently change.
