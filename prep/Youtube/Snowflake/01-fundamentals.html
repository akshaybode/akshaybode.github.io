1. What is Snowflake?

Snowflake is a full data warehouse built for the cloud. Its vision: let every customer access all their data in one place to make actionable decisions anytime, anywhere, with any number of users. It's a "next-generation" warehouse built from the ground up for the cloud to solve today's data and analytical challenges.

The 3 defining features

① SQL Data WarehouseAnyone who knows SQL can use it
+
② Built for CloudRuns on AWS / Azure / GCP
+
③ Delivered as a ServiceNo infra setup or management
SaaS, not software you install. There's nothing to download. Snowflake runs entirely in the cloud and manages all hardware, software, upgrades, and tuning for you. You interact with it through a browser UI (Snowsight) or a SQL client.

  • To move data into the EDW you needed a separate ETL tool (Informatica, SSIS, …).
  • Source systems mostly produce structured data; unstructured/semi-structured data (from web apps, logs, etc.) had to land first in a data lake (S3, Azure Blob) before it could be used.
  • The EDW ran on Oracle / SQL Server / Teradata; data marts (smaller, department-specific subsets) ran on yet more engines.
  • The pain: many different tools and technologies to buy, integrate, secure, and tune — expensive, complex, and hard to scale.

  • Snowflake replaces the legacy landscape — EDW, data marts, and much of the data-lake role collapse into a single platform.
  • Because storage is cheap and compute is elastic, you can switch from ETL to ELT: load raw data first, then transform it inside Snowflake with SQL.
  • Data marts become logical (views / schemas over the same data) instead of separate physical systems.
  • Snowflake natively handles structured and semi-structured data (JSON, Avro, Parquet via the VARIANT type), so the separate data-lake step is often unnecessary.
One-line takeaway: the legacy world needed a different tool for every stage; Snowflake unifies ingestion, the warehouse, marts, and sharing into one cloud service.

5. A New Architecture for Data Warehousing

To appreciate Snowflake's design, compare the three classic ways to build a warehouse — multi-cluster, shared data, in the cloud is the one Snowflake pioneered.

Shared-disk vs Shared-nothing vs Multi-cluster shared data
Traditional

Shared-disk

Shared storage, single cluster. All compute nodes read one central disk. Simple data management, but the shared disk becomes a bottleneck and it's hard to scale.

Traditional

Shared-nothing (MPP)

Decentralized, local storage, single cluster. Each node owns a slice of the data — fast parallelism, but storage and compute are coupled, so scaling one means scaling both.

Snowflake

Multi-cluster, shared data

Centralized, scale-out storage + multiple independent compute clusters. One copy of the data, many warehouses reading it at once. Scale storage and compute separately.

Snowflake is a hybrid of the two traditional models: a central data repository like shared-disk (one source of truth), processed by massively parallel compute clusters like shared-nothing (each node works on part of the data) — but with compute clusters that are independent and elastic.

6. The Three-Layer Architecture

That hybrid design is implemented as three independent layers. This separation is the single most important idea in Snowflake — it's why it scales and bills the way it does.

The 3 layers (top → bottom)

☁️ Cloud Services Layer — "the brain"

Coordinates everything across the platform. Runs on Snowflake-managed compute.

Authentication & access Infrastructure mgmt Query parsing & optimization Metadata management Transactions Security Result cache

⚙️ Compute Layer — Virtual Warehouses ("the muscle")

One or more MPP clusters that actually execute queries. Each warehouse is independent and sized X-Small → 4X-Large. Many can run at once on the same data without competing.

🗄️ Storage Layer — "the disk" (centralized)

All your data stored compressed & columnar in cloud object storage (S3 / Blob / GCS) as micro-partitions. Snowflake fully manages organization, compression, and metadata.

The key idea — storage and compute are separated. Storage scales on its own (pay for what you store). Compute (virtual warehouses) scales on its own (pay only while a warehouse runs). Multiple warehouses can read the same data simultaneously without slowing each other down. This is the practical payoff of "multi-cluster, shared data."

Layer 1 — Database Storage

  • When you load data, Snowflake reorganizes it into its own optimized, compressed, columnar format.
  • Data is stored as micro-partitions — small (≈50–500 MB uncompressed) contiguous units of storage.
  • Snowflake keeps rich metadata per micro-partition (min/max values, distinct counts, etc.) so it can prune partitions and skip data that can't match a query — this is what makes it fast (no manual indexes).
  • You never access storage files directly — only through SQL.

Layer 2 — Compute (Virtual Warehouses)

  • A virtual warehouse is a cluster of compute resources (CPU, memory, temp storage) that runs queries.
  • Sizes go X-Small, Small, Medium, Large … 4X-Large — each size up roughly doubles the compute (and credits/hour).
  • Scale up (bigger warehouse) for heavy queries; scale out (multi-cluster, Enterprise+) for more concurrent users.
  • Warehouses can auto-suspend when idle (stop billing) and auto-resume when a query arrives.
  • Different teams can each get their own warehouse on the same data — no contention.
-- create a small warehouse that pauses after 60s idle
CREATE WAREHOUSE my_wh
  WITH WAREHOUSE_SIZE = 'XSMALL'
       AUTO_SUSPEND = 60
       AUTO_RESUME = TRUE
       INITIALLY_SUSPENDED = TRUE;

Layer 3 — Cloud Services

The coordination layer that ties storage and compute together. It runs on Snowflake-managed compute (you generally don't pay extra unless its usage is heavy relative to compute). It handles authentication & access control, infrastructure management, the query optimizer, metadata (which also powers result caching), transactions, and security.

Why this matters: because the brain (cloud services), muscle (compute), and disk (storage) are independent, Snowflake can cache results, scale a single team's queries instantly, and bill storage and compute separately — the core advantages you'll see repeatedly through the rest of the course.
