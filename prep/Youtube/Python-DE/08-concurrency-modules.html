⚡ Multi-threading, Requests & OS

The final, highest-impact tools: run work in parallel with threads, pull data from APIs with the requests module, and manage files with the os module.

Multi-threading: do many things at once

A normal loop runs in sequence — table 1 finishes, then table 2, then table 3. With 100 tables you wait for all 99 before the last starts. Multi-threading runs them in parallel.

sequential loop vs parallel threads
Sequentialt1 → t2 → t3 (slow)
vs
Parallelt1 + t2 + t3 together
DE win: processing 100 Spark tables that each do the same thing — run them all together for an almost 100× speed-up instead of waiting in a queue.

ThreadPoolExecutor

The best tool for multi-threading. random + time.sleep are used here to simulate tables of different sizes.

import time, random
from concurrent.futures import ThreadPoolExecutor

def my_function(p_x):
    wait = random.randint(1, 10)
    time.sleep(wait)                       # simulate processing time
    print(f"I am {p_x}, I took {wait} seconds")

tables = ["orders", "products", "customers", "reviews", "cancels"]

Approach 1 — map (creates all futures at once)

with ThreadPoolExecutor(max_workers=len(tables)) as executor:
    futures = executor.map(my_function, tables)

Approach 2 — submit (creates futures one by one)

with ThreadPoolExecutor(max_workers=len(tables)) as executor:
    for i in tables:
        future = executor.submit(my_function, i)
  • max_workers: set to len(tables) so each item gets its own worker, all running in parallel.
  • map: applies the function across the whole iterable simultaneously (the speaker's preference).
  • submit: the loop only assigns each value and releases it to a thread — it does not wait for completion.
MethodHow it works
executor.map(fn, iterable)Maps the function over every item, all at once
executor.submit(fn, item)Submits a single task; loop to submit many without blocking

The requests Module: calling APIs

As a data engineer you'll make many API calls. An API (Application Programming Interface) is the middleman between you and a third-party application's data — you can't connect directly.

ResponseJSON back to you
import requests

url = "https://jsonplaceholder.typicode.com/posts/1"
response = requests.get(url)        # GET request to fetch data

print(response.status_code)         # 200 = success (404 = not found)

data = response.json()              # parse JSON -> Python dict
print(data)
  • requests.get(url) fetches data from an endpoint.
  • .status_code: 200 = success, 404 = error, etc.
  • .json() converts the response into a usable dictionary (most APIs return JSON).
Note: every API is unique — always read the provider's documentation. Some online compilers don't ship the requests module; switch compilers if import requests fails.

The os Module: file management

The os (operating system) module works with your local machine — files and folders. Data engineers more often use cloud equivalents (Databricks dbutils, Azure notebookutils), but the concepts are identical, so learning os makes those a piece of cake.

import os

os.getcwd()                       # current working directory
os.makedirs("C:/Users/you/folder")# create folder(s)
os.listdir()                      # list files & folders
os.path.abspath("main.py")        # absolute path of a file
FunctionDoes
os.getcwd()Get the current working directory
os.makedirs(path)Create a folder (and parents)
os.listdir(path)List all files/folders in a directory
os.path.abspath(file)Get the absolute path of a file
Heads up: os only works on a real machine, not most online compilers. It's used heavily at deployment time (paths, environment variables).

✅ Course Wrap-up & Key Takeaways

  • Multi-threading with ThreadPoolExecutor runs tasks in parallel — huge for processing many tables; use map or submit.
  • requests calls APIs: .get(), check .status_code (200), parse with .json().
  • os handles local files; cloud dbutils / notebookutils build on the same ideas.
The big picture: you're a data engineer, not a web developer. You now have a strong base — basics, data structures, functions, exceptions, OOP, inheritance, concurrency and APIs. From here you can pick up any new module or concept quickly. These were the non-negotiables.
