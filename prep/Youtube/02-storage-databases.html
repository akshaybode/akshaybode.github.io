1. Data Generation & Storage

Data is generated from many sources, then stored in storage systems.

Sources → Storage
SourcesTransactional (RDBMS), IoT devices, web/social media, logs & machine data, APIs / 3rd-party
→
StorageRDBMS, NoSQL, Data Warehouse, Object Storage (Data Lake)
RDBMS is tricky: it's both generation (connected to the app) and storage (data lives there). From the DE point of view, treat RDBMS as a generation source; from the app point of view, it's a storage layer.

Object storage = S3, Google Cloud Storage, Azure Blob Storage. When used to dump raw data, it's called a data lake (covered in Part 3).

2. DBMS & SQL

A DBMS (Database Management System) is built to store data in a structured way so you can query it fast. You could use Excel/Google Sheets, but finding one record in millions/billions of rows would be impossible — DBMS is designed for exactly this workload.

Popular DBMS

  • PostgreSQL — open source, easiest to start
  • MySQL — open source
  • Microsoft SQL Server, Oracle — enterprise-level

SQL — Structured Query Language

The language used to communicate with a database. Core operations (CRUD):

SELECT * FROM Student WHERE id = 2;          -- read
INSERT INTO Student VALUES (3, 'Akash', 25, 'Delhi');  -- create
UPDATE Student SET age = 27 WHERE id = 1;     -- update
DELETE FROM Student WHERE id = 3;             -- delete

Data lives in tables with column names and rows of actual data:

1D26Mumbai
2Akash25Delhi

3. Data Modeling (Relational)

Data modeling is a visual representation of how your data looks and is stored in an RDBMS. Imagine you're building Amazon from scratch. What would you store?

Tables: Orders, Users, Products, Payments, Shipping, Sellers (real Amazon has hundreds).

E-commerce data model (relationships via keys)
Users
  • user_id (PK)
  • name
  • age
  • address
Orders
  • order_id (PK)
  • order_name
  • order_date
  • user_id (FK)
  • product_id (FK)
  • payment_id (FK)
Products
  • product_id (PK)
  • name, category
  • description, weight
  • quantity, unit_size
  • seller_id (FK)
Payments
  • payment_id (PK)
  • amount
  • payment_date
Shipping
  • shipping_id (PK)
  • shipping_date
  • order_id (FK)
Sellers
  • seller_id (PK)
  • seller_name
  • age, location
  • Primary Key (PK) — uniquely identifies each row (e.g., product_id).
  • Foreign Key (FK) — links one table to another (e.g., Orders.user_id → Users.user_id).
  • A user orders a product → Orders holds user_id + product_id; Shipping holds order_id; Products holds seller_id.

Every column has a schema / data type (e.g., order_id INT, order_name STRING, order_date DATE). Because it's strictly structured and queried with SQL, this is a SQL / relational database table. Tools like draw.io are used to draw real models.

4. NoSQL Databases

SQL stores data in rows & columns. NoSQL stores data in other formats for specific workloads.

NoSQL typeIdea
Key-ValueLike JSON/Python dict: {id:1, name:"D", age:26}. Search by key.
Column family / Wide columnData stored within columns (e.g., Cassandra).
DocumentDocument-oriented storage (e.g., MongoDB).
GraphUsed for representing relationships/networks.
Focus note: In the real world you'll mostly work with SQL databases and data warehouses, because most data is stored in SQL systems. NoSQL is good-to-know for specific use cases.

6. OLTP vs OLAP (Row vs Column)

The two data-processing systems a data engineer constantly juggles between.

As a Data Engineer, you move data OLTP → (transform) → OLAP
OLTPOnline Transactional Processing
Relational DB
→
Transformation
→
OLAPOnline Analytical Processing
Data Warehouse

OLTP — for transactions

  • Handles individual transactions: send money, buy a product, return an item.
  • Optimized for fast CRUD (Create, Read, Update, Delete) at the row level.
  • Row-based storage — every record stored together as a row.

OLAP — for analysis

  • Built for analytical workloads — e.g., analyze the last 5 years of data.
  • Column-based storage — each column stored together.

Why storage layout matters

Run SELECT SUM(payment) FROM users; on each system:

OLTP — Row-based ❌ slow for analytics

Stored row by row. To sum payment, it must scan every full row (id, name, age, payment) then pick one column — wasteful.

idnameagepay idnameagepay idnameagepay

OLAP — Column-based ✅ fast for analytics

Stored column by column. To sum payment, it goes straight to the payment column — no wasted scanning.

ididid namenamename paypaypay
OLTPOLAP
Full nameOnline Transactional ProcessingOnline Analytical Processing
StorageRow-basedColumn-based
Best forFast inserts/updates/reads at row levelAggregations & analysis over huge data
Example systemRelational DB (Postgres, MySQL)Data warehouse (Snowflake, BigQuery, Redshift)
Neither is "better": both have their place. OLTP is great for row-level transactions; OLAP is great for analysis. DE = take data from OLTP, transform, load into OLAP.
