08 — Apache Airflow

The orchestrator: schedules, sequences, retries, and monitors your pipeline tasks. "Cron for cron, plus dependencies, retries, backfills, and observability."

Core concepts

  • DAG (Directed Acyclic Graph): your workflow — tasks + dependencies, no cycles.
  • Task: one unit of work (an operator instance).
  • Operator: a template for a task — PythonOperator, BashOperator, SnowflakeOperator, S3ToSnowflakeOperator, DbtCloud/Bash for dbt, sensors.
  • Sensor: a task that waits for a condition (file arrives, partition ready).
  • Scheduler: parses DAGs, decides what to run when. Executor: how/where tasks run (Local, Celery, Kubernetes). Webserver: the UI.
  • DAG run: one execution for a given logical date (data interval).
  • Task instance: one task in one DAG run; states: queued, running, success, failed, up_for_retry, skipped.

A modern DAG (TaskFlow API)

from airflow.decorators import dag, task
import pendulum

@dag(schedule="0 2 * * *",                 # daily at 02:00
     start_date=pendulum.datetime(2025, 1, 1, tz="Asia/Kolkata"),
     catchup=False, max_active_runs=1,
     default_args={"retries": 2, "retry_delay": pendulum.duration(minutes=5)},
     tags=["billing"])
def billing_pipeline():

    @task
    def extract(**ctx):
        ds = ctx["ds"]                      # the data interval date
        return f"s3://lake/raw/invoices/dt={ds}/"

    @task
    def load_to_snowflake(path: str): ...

    @task
    def run_dbt(): ...                      # BashOperator: `dbt build`

    path = extract()
    load_to_snowflake(path) >> run_dbt()

billing_pipeline()

Scheduling & backfill (common confusion — explain clearly)

  • A run with logical/data date = the start of the interval; it executes after the interval ends. (Daily DAG for 2025-06-01 runs early on 2025-06-02.)
  • catchup=True: Airflow backfills all missed intervals since start_date. Set catchup=False unless you intentionally want historical backfills.
  • Backfill: rerun a date range to reprocess history (airflow dags backfill).
  • Use {{ ds }} (the data date) to make tasks process the right slice → enables reruns/backfills cleanly.

Idempotency & retries (the maturity signal)

  • Design each task so rerunning a date overwrites that date's slice (no dupes): partition overwrite or MERGE on keys.
  • Set sensible retries, retry_delay, execution_timeout, and alerting on failure.
  • Don't pass big data through XCom (it's for small metadata, stored in the DB); pass file paths/locations instead.

Dependencies & control flow

  • a >> b >> c (set order); [a, b] >> c (fan-in); branching via @task.branch.
  • Trigger rules: all_success (default), all_done, one_failed, etc.
  • Pools limit concurrency for shared resources; priority_weight orders tasks.

Connections, variables, secrets

  • Connections: store DB/cloud creds (Snowflake, S3) in the UI/secret backend.
  • Variables: small config values. Use a secrets backend (AWS Secrets Manager, Vault) in prod — never hardcode credentials.

Best practices (say these)

  1. Keep DAG files lightweight — no heavy work at parse time (top level).
  2. One clear responsibility per task; small, idempotent, retryable.
  3. Use the right tool for compute — Airflow orchestrates, it shouldn't crunch big data itself (push work to Spark/Snowflake/dbt).
  4. catchup=False by default; parameterize by ds for safe backfills.
  5. Monitor: SLAs, alerts on failure, sensor timeouts to avoid stuck DAGs.

Airflow vs alternatives (one-liners)

  • Airflow: code-first DAGs, huge ecosystem, the industry default.
  • Dagster / Prefect: newer, asset/dataflow-oriented, nicer local dev.
  • Snowflake Tasks / dbt Cloud scheduler: lighter, in-platform scheduling.
