07 — dbt (data build tool)

dbt is the T in ELT — the transformation layer on top of Snowflake. It brings software engineering (version control, tests, modularity, docs) to SQL.

Mental model

  • You write models = SELECT statements in .sql files. dbt wraps each in CREATE TABLE/VIEW AS and runs them in dependency order.
  • dbt does not extract or load — it transforms data already in the warehouse.

Core concepts

  • Model: one .sql file → one table/view. Built with dbt run.
  • ref(): reference another model → dbt builds the DAG and the right order. sql -- models/marts/fct_invoices.sql SELECT * FROM {{ ref('stg_invoices') }}
  • source(): reference raw tables loaded into the warehouse. sql SELECT * FROM {{ source('raw', 'invoices') }}
  • Materializations (how a model is built):
  • view (default, no storage), table (rebuilt each run), incremental (only new/changed rows), ephemeral (inlined CTE, not in warehouse).

Project layout (typical)

models/
  staging/    stg_*.sql      -- 1:1 with sources, cleaned/renamed/typed
  intermediate/ int_*.sql    -- reusable business logic
  marts/      dim_*, fct_*   -- star-schema, business-facing
dbt_project.yml

Incremental models (the important one)

{{ config(materialized='incremental', unique_key='invoice_id') }}
SELECT * FROM {{ source('raw','invoices') }}
{% if is_incremental() %}
  WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }})
{% endif %}
  • dbt builds full on first run, then only processes new rows; merges on unique_key.

Tests (data quality, version-controlled)

  • Generic (in YAML): not_null, unique, accepted_values, relationships (FK). ```yaml models:
    • name: fct_invoices columns:
      • name: invoice_id tests: [unique, not_null]
      • name: customer_id tests:
        • relationships: to: ref('dim_customer') field: customer_id ```
  • Singular: a .sql file in tests/ that should return 0 rows to pass.
  • Run with dbt test. CI fails the build if tests fail.

Snapshots (SCD Type 2, built in!)

{% snapshot scd_customers %}
{{ config(target_schema='snapshots', unique_key='customer_id',
          strategy='timestamp', updated_at='updated_at') }}
SELECT * FROM {{ source('raw','customers') }}
{% endsnapshot %}
  • dbt manages dbt_valid_from/dbt_valid_to automatically — instant SCD2 history.

Jinja & macros (DRY SQL)

  • dbt SQL is templated with Jinja ({{ }}, {% %}). Macros = reusable SQL functions. {{ var(...) }} and {{ env_var(...) }} for config.

Other essentials

  • Seeds: small CSVs version-controlled and loaded (dbt seed) — e.g. mappings.
  • Docs: dbt docs generate && dbt docs serve → auto lineage graph + descriptions.
  • Exposures / freshness: declare downstream BI; dbt source freshness checks SLAs.

Commands you should know

dbt run                 # build models
dbt test                # run tests
dbt build               # run + test + snapshot + seed, in DAG order
dbt run --select stg_invoices+   # this model and everything downstream
dbt snapshot

Why dbt (interview soundbite)

"dbt turns transformations into versioned, tested, documented SQL with automatic lineage. It replaces brittle stored-procedure spaghetti with a DAG I can test in CI, build incrementally, and hand to any analyst to read."
