05 — Data Warehousing, ETL/ELT, Lakehouse & File Formats

ETL vs ELT

  • ETL (extract → transform → load): transform before loading. Classic, on-prem, fixed-capacity engines (Informatica, your Amdocs batch world).
  • ELT (extract → load → transform): load raw into the warehouse/lake, transform inside it with SQL (Snowflake + dbt). Modern default — cheap storage + elastic compute.
  • Be ready to say: "We moved from ETL to ELT because the warehouse scales compute on demand, raw data stays replayable, and transforms live in version-controlled SQL."

Data Warehouse vs Data Lake vs Lakehouse

Warehouse Lake Lakehouse
Data Structured Any (raw files) Both
Schema On write On read Both
Cost Higher Cheap (object storage) Cheap storage + warehouse features
Tools Snowflake, Redshift, BigQuery S3/ADLS/GCS + Spark Delta Lake, Iceberg, Snowflake
Risk Rigid "Data swamp" Best of both
  • Lakehouse = lake storage + ACID table formats (Delta/Iceberg/Hudi) giving transactions, schema enforcement, time travel, upserts.

File formats (know columnar vs row)

Format Type Use
CSV/JSON Row, text Interchange/config; bad for analytics (no schema, big, slow)
Parquet Columnar, binary Default for analytics — compression + column pruning
ORC Columnar Hive/Hadoop ecosystems
Avro Row, binary Streaming/Kafka, schema evolution, record-by-record
- Columnar (Parquet) wins for analytics: read only needed columns, great
compression (similar values together), predicate pushdown via row-group stats.
- Row (Avro) wins for write-heavy/streaming where you need whole records.

Partitioning & clustering

  • Partitioning: split data by a column (e.g. dt=2025-06-01/) so queries scan only relevant partitions (partition pruning). Pick low-cardinality, commonly filtered columns (date, region). Too many tiny partitions = "small files problem."
  • Clustering / bucketing: order/group data within storage to speed filters/joins (Snowflake clustering keys; Spark bucketing).

Compression

  • Snappy (fast, default for Parquet), gzip (smaller, slower), zstd (good balance).

Incremental loading patterns

  • Full load: reload everything (simple, small tables).
  • Incremental / delta: load only new/changed rows since last watermark (updated_at > last_run). Use MERGE/upsert into the target.
  • CDC (Change Data Capture): capture inserts/updates/deletes from source logs (Debezium, Snowflake Streams). Best for low-latency, accurate sync.

Idempotency (a word interviewers love)

  • Re-running a job produces the same result, no duplicates. Achieve via: overwrite-by-partition, MERGE/upsert on keys, or delete-then-insert for the window.

Data quality

  • Tests: not-null, unique, accepted values, referential integrity, freshness, row-count vs source. Tools: dbt tests, Great Expectations. Fail loud, quarantine bad rows.

Orchestration & scheduling

  • A DAG of tasks with dependencies, retries, scheduling, backfills, alerting → Airflow (see 08). Don't cron a pile of scripts in prod.
