03 — PySpark (the #1 hard skill)

If you only master one new thing, make it this. Two parts: the DataFrame API (what you write) and internals (what gets you hired at mid-level).

Why Spark

  • Distributed, in-memory processing across a cluster. Handles data bigger than one machine's RAM. Lazy: builds a DAG, optimizes (Catalyst), runs on an action.

Core concepts

  • RDD (low-level) → DataFrame/Dataset (high-level, optimized — use these).
  • Transformations are lazy (select, filter, join, groupBy, withColumn).
  • Actions trigger execution (show, count, collect, write, take).
  • Partition = a chunk of data processed by one task. Stage = set of tasks between shuffles. Job = triggered by one action.

DataFrame API essentials

from pyspark.sql import SparkSession, functions as F, Window
spark = SparkSession.builder.appName("billing").getOrCreate()

df = spark.read.parquet("s3://lake/raw/invoices/")

out = (df
    .filter(F.col("amount") > 0)
    .withColumn("month", F.month("invoice_date"))
    .withColumn("amount_inr", F.col("amount") * F.lit(83))
    .select("customer_id", "month", "amount_inr"))

agg = (out.groupBy("customer_id", "month")
          .agg(F.sum("amount_inr").alias("total"),
               F.count("*").alias("n_invoices")))

agg.write.mode("overwrite").partitionBy("month").parquet("s3://lake/curated/")

Joins (and the interview trap: broadcast)

# Small dimension joined to a huge fact -> broadcast the small one (no shuffle)
from pyspark.sql.functions import broadcast
fact.join(broadcast(dim_customer), "customer_id", "left")
  • Join types: inner, left/right/full outer, left_semi, left_anti.
  • Shuffle join (sort-merge) is default for big-big; broadcast join avoids shuffle when one side is small (< ~10MB, tunable). Naming this is a green flag.

Window functions in Spark

w = Window.partitionBy("customer_id").orderBy(F.desc("invoice_date"))
latest = (df.withColumn("rn", F.row_number().over(w))
            .filter(F.col("rn") == 1)        # dedupe: keep latest per customer
            .drop("rn"))

Internals (this is where mid-level is won)

  • Shuffle: data redistributed across partitions (after groupBy, join, repartition). Expensive — network + disk. Minimize shuffles.
  • repartition(n) = full shuffle, can increase partitions / rebalance. coalesce(n) = merge partitions, no full shuffle, only reduces. Use coalesce before writing few files; repartition to fix skew/parallelism.
  • Caching: df.cache() / persist() when a DataFrame is reused multiple times.
  • AQE (Adaptive Query Execution): runtime optimization — coalesces shuffle partitions, switches join strategies, handles skew. On by default in Spark 3+.
  • Predicate/column pushdown: filters and column selection pushed to Parquet/ source so less data is read. Partitioned data → partition pruning.
  • Data skew: one key has way more rows → one task is the bottleneck. Fixes: enable AQE skew handling, salt the key, or broadcast the smaller side.

Spark SQL (you can lean on your SQL strength)

df.createOrReplaceTempView("invoices")
spark.sql("""
  SELECT customer_id, SUM(amount) total
  FROM invoices WHERE amount > 0 GROUP BY customer_id
""")

Read/write & schemas

schema = "customer_id STRING, amount DOUBLE, invoice_date DATE"
df = spark.read.schema(schema).csv("raw.csv", header=True)   # define schema (don't infer in prod)
df.write.format("delta").mode("append").save("/lake/curated") # Delta for ACID/upserts
  • Prefer explicit schema over inferSchema (faster, safer).
  • Write modes: append, overwrite, ignore, error.

Structured Streaming (conceptual — month 2)

  • Treat a stream as an unbounded table; micro-batches; checkpointing for exactly-once; watermarks for late data. Read Kafka → transform → write to sink.

Performance checklist (say these out loud)

  1. Read only needed columns/partitions (pushdown + pruning).
  2. Filter early; avoid wide transformations when possible.
  3. Broadcast small dimensions.
  4. Right-size partitions (~128MB each); coalesce before writing.
  5. Cache reused DataFrames; avoid collect() on big data.
  6. Use Parquet/Delta, not CSV/JSON, for processing.
  7. Watch skew; lean on AQE.
