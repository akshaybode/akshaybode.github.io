09 — Data Engineering System Design

The "design a pipeline for X" round. There's a repeatable structure — use it every time.

The framework (memorize this flow)

  1. Clarify requirements - Data sources & formats? Volume (GB/TB/day)? Velocity (batch vs streaming)? - Latency/freshness SLA (real-time? hourly? daily?) - Who consumes it (BI, ML, app) and what queries? - Accuracy/quality needs, retention, cost/budget, compliance (PII/GDPR).
  2. Sketch the high-level architecture (the 5 stages below).
  3. Pick technologies and justify each (don't name-drop — tie to requirements).
  4. Address the hard parts: scale, schema evolution, idempotency, failure recovery, data quality, monitoring, cost.
  5. Trade-offs & alternatives — show you considered options.

The 5 stages of (almost) every pipeline

1. Ingestion → 2. Storage → 3. Processing/Transform → 4. Serving → 5. Orchestration+Observability
  1. Ingestion: batch (Fivetran/Airbyte, COPY, JDBC) vs streaming (Kafka, Kinesis). Pull vs push; full vs incremental; CDC for DB sources.
  2. Storage: raw landing (object store: S3/ADLS) + warehouse (Snowflake). Lakehouse table format (Delta/Iceberg) if lake-centric. Partition by date.
  3. Processing: SQL/dbt in-warehouse (ELT) or Spark/PySpark for heavy/unstructured. Layer it (raw → staging → marts / bronze → silver → gold).
  4. Serving: BI tables/views, a semantic layer, reverse-ETL, or an API/feature store.
  5. Orchestration: Airflow DAGs; plus monitoring, alerting, lineage, data-quality tests.

Batch vs streaming (know when to use which)

Batch Streaming
Latency minutes–hours seconds
Complexity lower higher (state, ordering, exactly-once)
Cost cheaper higher
Use reporting, most analytics fraud, monitoring, real-time dashboards
- Default to batch unless the SLA truly needs streaming. Lambda (batch + speed
layers) vs Kappa (stream-only) architectures — name them.

Cross-cutting concerns (sprinkle these — they win points)

  • Idempotency: reruns don't duplicate (partition overwrite / MERGE on keys).
  • Incremental + watermark: process only new data; track high-water mark.
  • Schema evolution: handle new/changed columns (Parquet/Avro evolution, dbt contracts).
  • Data quality: tests (not-null/unique/freshness/row-count), quarantine bad rows, alert.
  • Failure recovery: retries, checkpoints, backfills, dead-letter queues.
  • Late/out-of-order data (streaming): watermarks, windowing.
  • Observability: logging, metrics, lineage, freshness SLAs, on-call alerts.
  • Cost: right-size compute, auto-suspend, partition pruning, avoid reprocessing.
  • Security/PII: encryption, masking, RBAC, least privilege, audit.

Worked example — "Design a pipeline for daily telecom billing analytics"

  • Requirements: ~50M invoice lines/day from Amdocs billing DB; daily freshness; BI + finance consumers; track customer plan changes historically.
  • Ingest: CDC from the billing DB (or daily incremental extract on updated_at) → land as Parquet in S3/ADLS partitioned by dt.
  • Load: COPY INTO Snowflake raw (or Snowpipe for continuous).
  • Transform (dbt): stg_invoices (clean/typed) → dim_customer (SCD2 snapshot), dim_plan, dim_date → fct_invoice (grain: one invoice line). Tests on keys.
  • Serve: marts for BI; aggregated revenue tables; semantic layer.
  • Orchestrate: Airflow daily DAG — extract → load → dbt build → quality gate → alert.
  • Concerns: idempotent partition overwrite; SCD2 for plan history; row-count reconciliation vs source; PII masking on customer data; auto-suspend warehouse for cost.

Other prompts to rehearse out loud

  • Ingest clickstream/events into a warehouse for analytics.
  • Build a near-real-time dashboard (Kafka → Spark Structured Streaming → sink).
  • Migrate on-prem Oracle/ETL (your Amdocs world!) to a cloud lakehouse.
  • Deduplicate and reconcile data from 3 systems with conflicting customer IDs.
  • Design a data-quality framework for a warehouse.
