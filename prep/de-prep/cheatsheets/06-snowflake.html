06 — Snowflake

Your chosen warehouse. Interviewers probe architecture, micro-partitions, warehouses (compute), performance, and cost. Know these.

Architecture (the headline)

Snowflake separates storage, compute, and cloud services — independently scalable. 1. Storage layer: data stored compressed, columnar, in micro-partitions on cloud object storage (S3/Azure Blob/GCS). 2. Compute layer: virtual warehouses = independent MPP compute clusters. Multiple warehouses read the same data without contention. 3. Cloud services layer: brains — query optimizer, metadata, security, transactions, result cache.

Soundbite: "Separation of storage and compute means I can scale a heavy transformation warehouse up without affecting the BI warehouse, and only pay for compute while it runs."

Micro-partitions & pruning (Snowflake's "indexing")

  • Data auto-divided into micro-partitions (~50–500MB uncompressed), columnar.
  • Snowflake stores metadata (min/max, distinct counts) per micro-partition.
  • Queries prune partitions using that metadata — no manual indexes needed.
  • Clustering key: for very large tables, define one so related rows co-locate → better pruning. Check with SYSTEM$CLUSTERING_INFORMATION. Don't over-cluster (costs credits).

Virtual warehouses (compute)

  • Sizes XS→4XL (each step ~2x compute & credits). Scale up (bigger) for complex queries; scale out (multi-cluster) for concurrency.
  • Auto-suspend (stop when idle) + auto-resume = cost control. Always set these.
  • Per-second billing, 60s minimum.

Caching (3 layers — a favorite question)

  1. Result cache (services layer): identical query within 24h, no compute used.
  2. Local disk / warehouse cache: data cached on warehouse SSD while running.
  3. Metadata cache: stats like COUNT(*), MIN/MAX answered from metadata.

Loading data

  • Bulk: COPY INTO from a stage (internal, or external S3/ADLS/GCS).
  • Continuous: Snowpipe (auto-ingest files as they land, serverless).
  • Streams + Tasks: STREAM captures CDC (changes on a table); TASK schedules SQL (e.g. MERGE the stream into a target) — Snowflake-native pipelines.
COPY INTO raw.invoices
FROM @my_s3_stage/invoices/
FILE_FORMAT = (TYPE = PARQUET)
MATCH_BY_COLUMN_NAME = CASE_INSENSITIVE;

Time Travel & cloning (differentiators)

  • Time Travel: query/restore data as of a past timestamp (default 1 day, up to 90 on Enterprise). SELECT ... AT (OFFSET => -3600); UNDROP TABLE.
  • Zero-copy cloning: CREATE TABLE dev CLONE prod; — instant, no storage copy (copy-on-write). Great for dev/test environments.
  • Fail-safe: 7-day Snowflake-managed recovery after Time Travel.

Table types

  • Permanent (Time Travel + Fail-safe), Transient (no Fail-safe, cheaper — good for staging), Temporary (session-only).

Semi-structured data

  • VARIANT column stores JSON/Avro/Parquet; query with col:path.to.field, FLATTEN() for arrays. Schema-on-read inside the warehouse.

Performance & cost levers (say these)

  • Right-size warehouse; auto-suspend aggressively; separate workloads into separate warehouses.
  • Use clustering only on huge, frequently-filtered tables.
  • Reduce scanned data: select columns, filter on clustered/partition-pruned columns.
  • Watch spilling to disk (warehouse too small) in the query profile.
  • Use result cache (don't add current_timestamp() that defeats it).
  • Materialized views for expensive repeated aggregations (costs maintenance credits).

RBAC basics

  • Role-based access; roles own grants; USAGE on warehouse/db/schema, SELECT on tables. System roles: ACCOUNTADMIN, SYSADMIN, SECURITYADMIN, PUBLIC.

Snowflake vs Redshift vs BigQuery (one-liners)

  • Snowflake: multi-cloud, true storage/compute separation, easy concurrency scaling.
  • Redshift: AWS-native, cluster-based (newer RA3 separates storage).
  • BigQuery: GCP serverless, pay-per-bytes-scanned, no warehouse sizing.
