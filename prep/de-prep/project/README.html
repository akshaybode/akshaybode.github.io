Portfolio Project — Telecom Billing Analytics Pipeline

Build this once. It becomes your answer to "tell me about a pipeline you built" in every interview, and it exercises the whole stack: PySpark + Parquet + Snowflake + dbt + Airflow. You already know the telecom billing domain — that's your unfair advantage.

Goal: a daily batch pipeline that turns raw telecom billing/CDR data into a dimensional model in Snowflake for revenue analytics, orchestrated by Airflow, transformed and tested by dbt.

Architecture

                        ┌─────────────────────────────────────────────┐
                        │              Airflow (orchestration)          │
                        └─────────────────────────────────────────────┘
  Synthetic billing  →  PySpark clean/      →  Parquet     →  Snowflake  →  dbt        →  BI marts
  data (generator)      conform (raw→stg)      (data lake)    (COPY raw)    (stg→dims/facts,
                                                                            tests, SCD2)

Stages (and the cheat sheet each maps to): 1. Generate synthetic source data — customers, plans, invoices/CDRs (Python). 2. Ingest/clean with PySpark → write partitioned Parquet ("data lake"). → 03, 05 3. Load Parquet into Snowflake RAW with COPY INTO. → 06 4. Transform with dbt: staging → dimensions (incl. SCD2) → fact. Tests. → 04, 07 5. Orchestrate the whole thing as a daily Airflow DAG (idempotent). → 08 6. Document: README + architecture diagram + sample analytics queries.

The data model (your talking point)

  • Grain of fct_invoice: one row per invoice line per billing cycle.
  • Dimensions: dim_customer (SCD2 — track plan/segment changes), dim_plan, dim_date, optionally dim_geography.
  • Measures: amount, tax, discount, usage_units; mostly additive.

Sample analytics it should answer

-- Monthly revenue by plan
SELECT d.month, p.plan_name, SUM(f.amount) revenue
FROM fct_invoice f
JOIN dim_date d  ON f.date_key = d.date_key
JOIN dim_plan p  ON f.plan_key = p.plan_key
GROUP BY 1, 2 ORDER BY 1, 2;

-- Top 10 customers by lifetime revenue
SELECT c.customer_name, SUM(f.amount) lifetime_value
FROM fct_invoice f JOIN dim_customer c ON f.customer_key = c.customer_key
WHERE c.is_current
GROUP BY 1 ORDER BY 2 DESC LIMIT 10;

-- Revenue impact of plan changes (uses SCD2 history)
SELECT plan_name, COUNT(*) customers, SUM(amount) revenue
FROM fct_invoice f JOIN dim_customer c ON f.customer_key = c.customer_key
GROUP BY 1;

Build order (fits Week 3)

  1. Day 1 — generate_data.py: write ~100k invoices across customers/plans/dates to CSV/JSON.
  2. Day 2 — clean_invoices.py (PySpark): dedupe, validate, cast, write Parquet partitioned by dt.
  3. Day 3 — Snowflake: create RAW tables, stage, COPY INTO. Verify counts.
  4. Day 4 — dbt: staging models + dim_*/fct_invoice + tests; add scd_customers snapshot.
  5. Day 5 — Airflow DAG wiring all steps; make reruns idempotent (overwrite by dt).
  6. Day 6 — README + architecture diagram + the analytics queries + a screenshot.

Make it interview-gold

  • Idempotency: rerunning a date overwrites that date's partition / MERGEs on keys — no dupes.
  • Data quality: dbt tests on keys + a row-count reconciliation (source vs warehouse).
  • SCD2: be ready to explain exactly how plan-change history is preserved.
  • Scale story: "At 100x I'd switch to CDC ingestion, cluster the fact on date, size the warehouse per workload, and parallelize Spark by partition."
  • Cost story: auto-suspend warehouse, partition pruning, incremental dbt models.

Free tools to build it for ₹0

  • Snowflake: 30-day free trial ($400 credits) — plenty for this.
  • Spark: local PySpark (pip install pyspark) — no cluster needed.
  • Airflow: pip install apache-airflow locally, or Astronomer's free local CLI (astro dev).
  • dbt: pip install dbt-snowflake.
  • Data: pip install faker to generate realistic synthetic billing data.
  • Put it all on GitHub with a clear README — recruiters and interviewers look.
