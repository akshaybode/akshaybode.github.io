Q&A Bank — Data Engineer (mid-level, Snowflake + dbt + Airflow)

~90 likely questions with tight model answers. Practice saying each in 2–4 sentences. Pair with the cheat sheets for depth.

A. SQL

1. GROUP BY vs PARTITION BY? GROUP BY collapses rows into one per group; PARTITION BY keeps all rows and adds a per-group calculation via a window function.

2. RANK vs DENSE_RANK vs ROW_NUMBER? ROW_NUMBER gives a unique sequential number. RANK leaves gaps after ties (1,1,3); DENSE_RANK doesn't (1,1,2). Use ROW_NUMBER to dedupe.

3. How do you find the 2nd highest value per group? A window function: DENSE_RANK() OVER (PARTITION BY grp ORDER BY val DESC) then filter rank = 2.

4. How do you deduplicate keeping the latest record? ROW_NUMBER() OVER (PARTITION BY key ORDER BY updated_at DESC) in a CTE, keep rn = 1.

5. WHERE vs HAVING? WHERE filters rows before aggregation; HAVING filters after GROUP BY, on aggregates.

6. What is a window frame (ROWS BETWEEN)? It defines which rows the window function sees relative to the current row — e.g. running totals (UNBOUNDED PRECEDING → CURRENT ROW) or moving averages (N PRECEDING).

7. CTE vs subquery vs temp table? CTEs improve readability and chaining; recursive CTEs handle hierarchies. Subqueries inline. Temp tables persist within a session and can help break up huge queries.

8. How would you optimize a slow query? Read the execution plan, filter early, select only needed columns, ensure partition pruning/clustering, avoid functions on filter columns, pre-aggregate before joins, and watch for skew and spilling.

9. MERGE / upsert — what and why? MERGE inserts new rows and updates changed ones in one statement against a key — the foundation of idempotent incremental loads.

10. NOT IN vs NOT EXISTS gotcha? NOT IN with a NULL in the subquery returns no rows; prefer NOT EXISTS (NULL-safe) for anti-joins.

11. How to write sessionization in SQL? Use LAG to find the gap from the previous event, flag a new session when the gap exceeds a threshold, then a running SUM of those flags as the session id.

12. Additive vs semi-additive vs non-additive facts? Additive sum across all dimensions (revenue); semi-additive can't sum across time (balances); non-additive are ratios you must recompute.

B. Python for DE

13. List vs generator — when use which? A list holds everything in memory (eager); a generator yields lazily, constant memory — use generators to stream large files/datasets.

14. When pandas vs PySpark? pandas for data that fits in one machine's memory; PySpark when data is distributed/larger than RAM or you need cluster parallelism.

15. How do you process a 50GB file in Python? Stream it line-by-line / in chunks with a generator (or pandas chunksize), or push it to Spark — never load it all into memory.

16. *args and kwargs?** *args collects positional args into a tuple; **kwargs collects keyword args into a dict — for flexible function signatures.

17. How do you make a pipeline script production-ready? Idempotent logic, structured logging, explicit error handling with retries, config via env vars (no hardcoded secrets), unit tests on transforms, and clear exit codes.

18. GIL — does it affect data engineering? The GIL limits CPU-bound threading; use multiprocessing for CPU work or, better, push heavy compute to Spark/the warehouse. I/O-bound work (API/DB) parallelizes fine with threads/async.

19. How do you handle secrets/credentials? Environment variables or a secrets manager (Vault, AWS Secrets Manager, Airflow connections) — never hardcode or commit them.

C. Spark / PySpark

20. Transformation vs action? Transformations (select, filter, join) are lazy and build a DAG; actions (count, collect, write) trigger execution.

21. What is a shuffle and why is it expensive? Redistributing data across partitions (after join/groupBy/repartition) over the network and disk — it's the main performance cost; minimize it.

22. repartition vs coalesce? repartition does a full shuffle and can increase or rebalance partitions; coalesce merges partitions without a full shuffle and only reduces them — use coalesce before writing fewer files.

23. What is a broadcast join? When one table is small, Spark ships it to every executor so the join needs no shuffle — ideal for large-fact-to-small-dimension joins.

24. How do you handle data skew? Enable AQE skew handling, salt the skewed key, broadcast the smaller side, or repartition on a better key.

25. What is AQE? Adaptive Query Execution — Spark 3 runtime optimization that coalesces shuffle partitions, switches join strategies, and handles skew based on actual stats.

26. cache/persist — when? When a DataFrame is reused across multiple actions, to avoid recomputing the lineage. Choose storage level by memory vs disk.

27. Why explicit schema over inferSchema? inferSchema scans data (slow) and can guess wrong; explicit schema is faster and deterministic in production.

28. RDD vs DataFrame? DataFrames are higher-level, columnar, and optimized by Catalyst/Tungsten; use them. RDDs are low-level and used only for fine control.

29. How does Spark decide partitions? By input splits (file/block size), spark.sql.shuffle.partitions after shuffles, and AQE coalescing. Aim ~128MB per partition.

30. predicate pushdown / partition pruning? Pushing filters and column selection down to the data source (Parquet) so Spark reads less; pruning skips entire partitions that don't match the filter.

31. Narrow vs wide transformation? Narrow: each input partition maps to one output partition, no shuffle (map, filter). Wide: requires shuffle across partitions (groupBy, join).

D. Data Modeling

32. Star vs snowflake schema? Star has denormalized dimensions one join from the fact (simpler, faster); snowflake normalizes dimensions into sub-tables (less storage, more joins).

33. Fact vs dimension table? Facts hold numeric measures and foreign keys at a defined grain; dimensions hold descriptive context (who/what/where/when).

34. What is the grain of a fact table? What a single row represents (e.g. one invoice line per billing cycle). Define it first — everything else follows.

35. Explain SCD Type 1 vs Type 2. Type 1 overwrites with no history; Type 2 adds a new versioned row with effective_from/to and is_current to preserve history.

36. How do you implement SCD Type 2? Detect changes via a hash of tracked attributes; close the current row (set effective_to/is_current=false) and insert a new current row. dbt snapshots automate this.

37. Surrogate vs natural key? Natural key comes from the source; surrogate is a warehouse-generated key used as the fact FK so SCD2 history and source changes don't break joins.

38. Conformed dimension? A dimension shared across multiple fact tables (e.g. date, customer), enabling consistent cross-process analysis.

39. Types of fact tables? Transaction (one row per event), periodic snapshot (state per period), accumulating snapshot (one row per process, updated through its lifecycle).

40. Normalization vs denormalization — when each? Normalize OLTP source systems for integrity; denormalize analytics models for read speed.

41. Kimball vs Inmon? Kimball is bottom-up dimensional marts (fast delivery); Inmon is a top-down normalized enterprise warehouse feeding marts.

E. Warehousing / ETL / Formats

42. ETL vs ELT? ETL transforms before loading (classic, fixed engines); ELT loads raw then transforms inside an elastic warehouse with SQL (Snowflake + dbt) — the modern default.

43. Data lake vs warehouse vs lakehouse? Lake = cheap raw storage, schema-on-read; warehouse = structured, schema-on-write, fast SQL; lakehouse = lake storage plus ACID table formats (Delta/Iceberg) giving warehouse features.

44. Why Parquet over CSV? Columnar + compressed + typed: read only needed columns, far smaller, supports predicate pushdown — much faster and cheaper for analytics.

45. Row vs columnar format? Columnar (Parquet/ORC) is best for analytics scans/aggregations; row (Avro/CSV) is best for write-heavy/streaming and whole-record access.

46. What is partitioning and the small-files problem? Splitting data by a column so queries scan only relevant slices (pruning); too many tiny partitions/files hurts performance and metadata — compact them.

47. Full vs incremental load? Full reloads everything (simple, small tables); incremental loads only new/changed rows since a watermark and merges them — scalable.

48. What is CDC? Change Data Capture streams inserts/updates/deletes from source logs (Debezium, Snowflake Streams) for low-latency, accurate sync.

49. What does idempotent mean for a pipeline? Re-running produces the same result with no duplicates — achieved via partition overwrite or MERGE on keys.

50. How do you ensure data quality? Automated tests (not-null, unique, accepted values, referential integrity, freshness, row-count reconciliation), quarantine bad rows, and alert on failures.

F. Snowflake

51. Explain Snowflake's architecture. Three separated layers: cloud storage (columnar micro-partitions), independent compute (virtual warehouses), and cloud services (optimizer, metadata, security). Storage and compute scale independently.

52. What are micro-partitions? Automatic ~50–500MB columnar storage units with per-partition metadata (min/max) enabling pruning — Snowflake's substitute for indexes.

53. What is a virtual warehouse? An independent MPP compute cluster you size (XS–4XL) and can scale up (bigger) or out (multi-cluster for concurrency); auto-suspend/resume controls cost.

54. Explain Snowflake caching. Result cache (identical query within 24h, no compute), warehouse local disk cache (data during runtime), and metadata cache (COUNT/MIN/MAX).

55. Clustering keys? For very large tables, they co-locate related rows to improve pruning; use selectively since reclustering costs credits.

56. How do you load data into Snowflake? Bulk COPY INTO from internal/external stages, continuous via Snowpipe, or Streams + Tasks for CDC-style native pipelines.

57. Time Travel and zero-copy cloning? Time Travel queries/restores past data (default 1 day, up to 90); zero-copy clone creates instant copy-on-write copies for dev/test without duplicating storage.

58. How do you control Snowflake cost? Right-size warehouses, aggressive auto-suspend, separate workloads, reduce scanned data via pruning/clustering, leverage result cache, and watch for disk spilling.

59. How does Snowflake handle semi-structured data? VARIANT columns store JSON/Avro/Parquet; query with path notation and FLATTEN for arrays (schema-on-read).

60. Transient vs permanent table? Transient tables skip Fail-safe and have shorter Time Travel (cheaper) — good for staging; permanent tables have full recovery.

61. Snowflake vs Redshift vs BigQuery? Snowflake: multi-cloud, true storage/compute separation, easy concurrency. Redshift: AWS cluster-based. BigQuery: serverless, pay-per-bytes-scanned.

G. dbt

62. What is dbt and what does it do? The transformation (T) layer of ELT — you write SELECT models and dbt builds them as tables/views in dependency order, with tests, docs, and lineage.

63. ref() vs source()? ref() references another dbt model (builds the DAG and order); source() references raw tables loaded into the warehouse.

64. dbt materializations? view (default), table (rebuilt each run), incremental (process only new/changed rows), ephemeral (inlined CTE, not persisted).

65. How do incremental models work? First run builds full; subsequent runs filter to new rows (e.g. updated_at > max) inside is_incremental() and merge on a unique_key.

66. dbt tests? Generic tests in YAML (not_null, unique, accepted_values, relationships) and singular tests (SQL returning 0 rows to pass); run with dbt test, fail the build in CI.

67. dbt snapshots? Built-in SCD Type 2 — dbt tracks history with dbt_valid_from/to using a timestamp or check strategy.

68. Why dbt over stored procedures? Version control, modularity via ref(), automated tests, auto-generated lineage and docs, and environment promotion — software engineering for SQL.

69. What is dbt build? Runs models, tests, snapshots, and seeds together in DAG order — the standard CI command.

70. Seeds and macros? Seeds are small version-controlled CSVs loaded into the warehouse; macros are reusable Jinja-templated SQL functions for DRY code.

H. Airflow / Orchestration

71. What is a DAG? A Directed Acyclic Graph of tasks with dependencies and no cycles — your workflow definition.

72. Operator vs task vs sensor? An operator is a task template (PythonOperator, etc.); a task is an operator instance; a sensor waits for a condition (file/partition ready).

73. What is catchup and why set it False? catchup=True backfills every missed interval since start_date; set False to avoid an accidental flood of historical runs.

74. logical/data date vs execution time? The logical date is the start of the data interval; the DAG actually runs after the interval ends, so a daily job for 2025-06-01 runs on 2025-06-02.

75. How do you make Airflow tasks idempotent? Parameterize by the data date ({{ ds }}) and overwrite that slice (partition overwrite or MERGE) so reruns/backfills don't duplicate data.

76. What is XCom and its limitation? A mechanism to pass small metadata between tasks, stored in the metadata DB — don't pass large data through it; pass file paths/locations instead.

77. How do you handle failures? retries with retry_delay, execution_timeout, alerting on failure, sensors with timeouts, and idempotent reruns/backfills.

78. Executors? Local (single machine), Celery (distributed workers), Kubernetes (pod per task) — choose by scale.

79. Should Airflow process the data itself? No — Airflow orchestrates; heavy compute is pushed to Spark/Snowflake/dbt. Keep tasks light and DAG parsing fast.

80. Airflow vs Snowflake Tasks vs dbt Cloud scheduler? Airflow is code-first, general, large ecosystem; Snowflake Tasks/dbt Cloud are lighter, in-platform scheduling for simpler needs.

I. System Design

81. How would you design a daily batch analytics pipeline? Clarify volume/SLA/consumers; ingest incrementally to object storage (Parquet, partitioned by date); COPY to Snowflake raw; transform with dbt (staging → dims/facts) with tests; serve marts; orchestrate with Airflow; add quality gates, monitoring, and idempotency.

82. Batch vs streaming — how do you choose? By latency SLA: batch for minutes–hours (cheaper, simpler), streaming for seconds (fraud, monitoring) at higher complexity/cost. Default to batch unless required.

83. How do you handle schema evolution? Use formats/contracts that tolerate change (Parquet/Avro evolution, dbt contracts), additive changes where possible, and validation to catch breaking changes early.

84. How do you guarantee no duplicates / exactly-once? Idempotent writes (MERGE/partition overwrite) for batch; checkpointing + idempotent sinks for streaming; dedupe on keys.

85. How would you migrate on-prem Oracle ETL to the cloud? Inventory sources/dependencies, land raw via CDC/extracts to object storage, rebuild transforms as dbt models, validate with row-count/value reconciliation, run parallel (old + new) before cutover.

86. How do you monitor a pipeline? Logs/metrics per task, data-quality tests, freshness SLAs, row-count reconciliation, lineage, and alerting/on-call for failures.

87. Lambda vs Kappa architecture? Lambda has separate batch and speed layers reconciled together; Kappa is stream-only with reprocessing from the log — simpler if streaming covers all needs.

J. Behavioral

88. Why move into data engineering? I already build and optimize data-heavy systems over high-volume telecom data; I want to do it as my core role with the modern stack (Spark, Snowflake, dbt, Airflow), which I've been learning and applying in projects.

89. Tell me about a hard technical problem you solved. (Use a STAR story — e.g. diagnosing a slow Amdocs batch/SQL job, finding the root cause, optimizing it, and quantifying the improvement.)

90. You haven't used [tool] in production — how do you handle that? Be honest, then bridge: "I built [project] with it and understand the concepts; here's how I'd apply it in your environment" — leading with transferable depth.

91. Why are you open to leaving your current company? Keep it factual and forward-looking: pursuing the dedicated DE direction I've been building toward; my company is restructuring. No negativity.
