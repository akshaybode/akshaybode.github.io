1. Spark Architecture Always Asked

The Big Picture

# Spark cluster components
#
#  ┌─────────────────────────────────────────────────┐
#  │  Driver Program  (your script + SparkSession)    │
#  │  - Creates SparkContext                          │
#  │  - Builds execution plan (DAG)                   │
#  │  - Schedules tasks                               │
#  └────────────────────┬────────────────────────────┘
#                       │
#  ┌────────────────────▼────────────────────────────┐
#  │  Cluster Manager (YARN / K8s / Standalone / Mesos)│
#  │  - Allocates resources                            │
#  └────────────────────┬────────────────────────────┘
#                       │
#         ┌─────────────┼─────────────┐
#         ▼             ▼             ▼
#   ┌─────────┐   ┌─────────┐   ┌─────────┐
#   │Executor │   │Executor │   │Executor │
#   │ - Tasks │   │ - Tasks │   │ - Tasks │
#   │ - Cache │   │ - Cache │   │ - Cache │
#   └─────────┘   └─────────┘   └─────────┘
#       (Worker Nodes — run Python/JVM tasks)

Key Concepts

TermMeaning
DriverYour main process. Coordinates execution. Holds SparkSession.
ExecutorJVM process on worker that runs tasks and caches data.
Cluster ManagerAllocates resources (YARN/K8s/Standalone).
JobOne Action triggers one Job (e.g., collect(), write).
StageA Job is split into Stages at shuffle boundaries.
TaskUnit of work — one task per partition per stage.
PartitionLogical chunk of data distributed across executors.
ShuffleData movement between executors (expensive!).

Transformations vs Actions

# TRANSFORMATIONS (lazy — build a DAG, don't execute):
#   select, filter, where, groupBy, join, orderBy,
#   withColumn, drop, distinct, union, repartition

# ACTIONS (eager — trigger execution):
#   show, count, collect, take, first, write, toPandas

df = spark.read.parquet("...")     # transformation (no execution!)
df2 = df.filter("age > 30")        # transformation
df3 = df2.select("name", "age")    # transformation
df3.show()                         # ACTION — NOW it runs the whole chain
Why lazy evaluation matters: Spark optimizes the entire chain before executing. Multiple filters get combined, columns get pruned, predicates get pushed down to source. This is why you should use DataFrame API over RDDs.

RDD vs DataFrame vs Dataset

RDDDataFrameDataset
Type SafetyYes (compile)No (runtime)Yes (Scala/Java only)
OptimizationNone (Spark can't see inside)Catalyst optimizer (best)Catalyst
PerformanceSlowerFastFast
APILow-level (map/filter)SQL-likeTyped SQL
PySparkAvailable but rarely usedPrimary API ✅Not available (Python is dynamic)
Rule: Use DataFrame API for everything in PySpark. Touch RDDs only for very specialized cases (rare).

Catalyst & Tungsten

  • Catalyst: Spark's query optimizer. Analyzes logical plan, applies rules (predicate pushdown, column pruning, constant folding) to produce optimized physical plan.
  • Tungsten: Execution engine that uses memory-efficient binary format and code generation for speed.
  • AQE (Adaptive Query Execution): Re-optimizes during runtime — coalesces partitions, switches join strategies based on actual data sizes. Enabled by default in Spark 3.x.

Write to Sinks

# Parquet (preferred!)
df.write.mode("overwrite").parquet("s3://bucket/out/")
df.write.mode("append").parquet("s3://bucket/out/")
# Modes: append, overwrite, ignore, error (default)

# Partitioned (creates folder structure: year=2026/month=05/)
df.write.partitionBy("year", "month").parquet("s3://bucket/out/")

# Delta
df.write.format("delta").mode("overwrite").save("s3://bucket/delta/")

# Save as table
df.write.mode("overwrite").saveAsTable("db.tablename")

# CSV (avoid for big data — slow!)
df.write.option("header", "true").csv("out/")

# Coalesce to single file (use sparingly — collects to one task)
df.coalesce(1).write.parquet("out/")

5. Joins & Optimization Critical Interview Topic

Join Types

df1.join(df2, on="customer_id", how="inner")
df1.join(df2, on="customer_id", how="left")          # left_outer
df1.join(df2, on="customer_id", how="right")
df1.join(df2, on="customer_id", how="outer")         # full_outer
df1.join(df2, on="customer_id", how="left_semi")     # like EXISTS — keeps left columns only
df1.join(df2, on="customer_id", how="left_anti")     # NOT EXISTS — left rows with no match
df1.crossJoin(df2)                                    # cartesian — careful!

# Different column names
df1.join(df2, df1.cust_id == df2.id, "inner")

# Multiple keys
df1.join(df2, on=["customer_id", "date"], how="inner")

# Complex join condition
df1.join(df2, (df1.id == df2.id) & (df1.date >= df2.start), "left")

Join Strategies

StrategyWhen Spark uses itBest for
Broadcast Hash JoinOne side < spark.sql.autoBroadcastJoinThreshold (default 10MB)Small × Big (e.g., lookup tables)
Sort-Merge JoinBoth sides large, sortableDefault for big × big
Shuffle Hash JoinOne side moderate, can fit per-partition hashRare; usually SMJ better
Broadcast Nested LoopCross / non-equi joinsAvoid if possible — slow

Broadcast Hint (Most Important Optimization)

from pyspark.sql.functions import broadcast

# Force broadcast of small DataFrame
big_df.join(broadcast(small_df), on="key", how="inner")

# Check threshold (default 10MB)
spark.conf.get("spark.sql.autoBroadcastJoinThreshold")

# Increase if needed
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 100 * 1024 * 1024)  # 100MB
Rule of thumb: Always broadcast the smaller side when one is <100MB. Eliminates shuffle on the bigger DataFrame — massive speedup.

Skew Handling

# Problem: some join keys have way more rows than others (e.g., NULL keys, 1 user with 1M events)
# → one partition takes hours while others finish in seconds

# Solution 1: Enable AQE skew handling (Spark 3.x — default ON)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Solution 2: Filter out null keys before join
big_df.filter(col("key").isNotNull()).join(other_df, "key")

# Solution 3: Salting (manual technique for non-AQE)
# Add a random suffix to skewed keys to spread the load
from pyspark.sql.functions import concat, lit, rand, floor

# Replicate small side for each salt
N = 10
small_salted = small_df.withColumn("salt", F.explode(F.array([F.lit(i) for i in range(N)])))
small_salted = small_salted.withColumn("salted_key", F.concat(col("key"), lit("_"), col("salt")))

# Add random salt to big side
big_salted = big_df.withColumn("salt", F.floor(F.rand() * N))
big_salted = big_salted.withColumn("salted_key", F.concat(col("key"), lit("_"), col("salt")))

result = big_salted.join(small_salted, "salted_key").drop("salt", "salted_key")

Common Join Pitfalls

  • Cartesian (cross) join by mistake — make sure ON condition is correct
  • NULL keys — NULLs never match in equi-joins, can also cause skew
  • Duplicate keys — multiplies rows; deduplicate first or use distinct
  • No broadcast for small dim tables — adds unnecessary shuffle
  • Joining before filtering — filter first to reduce shuffle volume

Best Practices

# 1. Filter before join (predicate pushdown)
small = orders.filter(col("date") >= "2026-01-01")
result = small.join(customers, "customer_id")

# 2. Select only needed columns (column pruning)
slim_orders = orders.select("customer_id", "amount", "date")

# 3. Broadcast small dimension
result = orders.join(broadcast(customers), "customer_id")

# 4. Avoid joining nullable keys without filter
orders.filter(col("customer_id").isNotNull()).join(customers, "customer_id")

# 5. Use bucketing for repeated joins on same key
orders.write.bucketBy(50, "customer_id").saveAsTable("orders_bucketed")

6. Window Functions

Window Spec

from pyspark.sql import Window
from pyspark.sql import functions as F

w = Window.partitionBy("region").orderBy(F.desc("sales"))

# Frame: ROWS BETWEEN ...
w_rolling = (Window.partitionBy("region")
             .orderBy("date")
             .rowsBetween(-6, 0))           # last 7 rows (rolling 7-day)

w_running = (Window.partitionBy("region")
             .orderBy("date")
             .rowsBetween(Window.unboundedPreceding, Window.currentRow))

w_full = (Window.partitionBy("region")
          .orderBy("date")
          .rowsBetween(Window.unboundedPreceding, Window.unboundedFollowing))

Ranking Functions

w = Window.partitionBy("region").orderBy(F.desc("sales"))

df = df.withColumn("row_num", F.row_number().over(w))      # 1,2,3 unique
df = df.withColumn("rank", F.rank().over(w))               # 1,2,2,4 (gaps)
df = df.withColumn("dense_rank", F.dense_rank().over(w))   # 1,2,2,3 (no gaps)
df = df.withColumn("percentile", F.percent_rank().over(w)) # 0..1
df = df.withColumn("ntile", F.ntile(4).over(w))            # 1..4 buckets

LAG / LEAD

w = Window.partitionBy("customer_id").orderBy("order_date")

df = df.withColumn("prev_amount", F.lag("amount", 1).over(w))
df = df.withColumn("next_amount", F.lead("amount", 1).over(w))
df = df.withColumn("amount_change", col("amount") - F.lag("amount", 1).over(w))

# With default value
df = df.withColumn("prev_amount", F.lag("amount", 1, 0).over(w))

Aggregate as Window

w = Window.partitionBy("region").orderBy("date").rowsBetween(-6, 0)

df = (df
  .withColumn("rolling_7day_avg", F.avg("sales").over(w))
  .withColumn("running_total", F.sum("sales").over(
      Window.partitionBy("region").orderBy("date")
            .rowsBetween(Window.unboundedPreceding, Window.currentRow)))
  .withColumn("region_total", F.sum("sales").over(Window.partitionBy("region")))
  .withColumn("pct_of_region", col("sales") / col("region_total"))
)

FIRST_VALUE / LAST_VALUE

w = (Window.partitionBy("product").orderBy("date")
     .rowsBetween(Window.unboundedPreceding, Window.unboundedFollowing))

df = df.withColumn("first_price", F.first("price").over(w))
df = df.withColumn("last_price", F.last("price").over(w))

Common Patterns

Top-N per group

w = Window.partitionBy("region").orderBy(F.desc("sales"))
top3 = df.withColumn("rn", F.dense_rank().over(w)).filter(col("rn") <= 3)

Deduplication (keep latest)

w = Window.partitionBy("id").orderBy(F.desc("updated_at"))
dedup = df.withColumn("rn", F.row_number().over(w)).filter(col("rn") == 1).drop("rn")

Sessionization (gaps & islands)

w = Window.partitionBy("user_id").orderBy("event_time")

df = (df
  .withColumn("prev_time", F.lag("event_time").over(w))
  .withColumn("gap_minutes",
      (F.unix_timestamp("event_time") - F.unix_timestamp("prev_time")) / 60)
  .withColumn("new_session",
      F.when(col("prev_time").isNull() | (col("gap_minutes") > 30), 1).otherwise(0))
  .withColumn("session_id", F.sum("new_session").over(w))
)
Performance: Each window function triggers a shuffle. Reuse same window spec when possible. Avoid orderBy without partitionBy — entire data goes to one partition!

7. Partitioning & File Formats

Partitions — In-Memory vs On-Disk

TypeWherePurpose
Partitions (in-memory)RDD/DataFrame splits across executorsParallelism unit
Partitions (on-disk)partitionBy folders (year=2026/month=05/)Partition pruning at read time
BucketsFiles within a folder split by hashAvoid shuffle for repeated joins

In-Memory Partition Operations

df.rdd.getNumPartitions()              # check current
df = df.repartition(200)               # full shuffle to N partitions
df = df.repartition("customer_id")     # hash partition by key
df = df.repartition(50, "customer_id") # both
df = df.coalesce(10)                   # reduce partitions WITHOUT full shuffle

# When to use which:
# - repartition: increase parallelism, distribute evenly (shuffle)
# - coalesce: decrease partitions cheaply (no full shuffle), before write
# - Default partition size target: 100-200 MB

On-Disk Partitioning (partitionBy)

# Write partitioned by date
df.write.partitionBy("year", "month").parquet("s3://bucket/data/")

# Folder structure:
#   s3://bucket/data/year=2026/month=01/part-00000.parquet
#   s3://bucket/data/year=2026/month=02/part-00000.parquet

# Reading with partition filter → Spark reads ONLY relevant folders
df = spark.read.parquet("s3://bucket/data/")

# - Each partition should have at least 100MB of data
# - Don't over-partition (small files problem)

8. User Defined Functions (UDFs)

Important: Python UDFs are SLOW because data must serialize between JVM ↔ Python for every row. Use built-in pyspark.sql.functions whenever possible. Only use UDFs when there's no native function. For better performance, use Pandas UDFs (vectorized).

Regular Python UDF (avoid when possible)

from pyspark.sql.functions import udf
from pyspark.sql.types import StringType, IntegerType

# Define
def categorize(age):
    if age is None: return "unknown"
    if age < 18: return "minor"
    if age < 60: return "adult"
    return "senior"

# Register
categorize_udf = udf(categorize, StringType())

# Use
df = df.withColumn("category", categorize_udf(col("age")))

# Decorator style
@udf(returnType=IntegerType())
def double(x):
    return x * 2 if x is not None else None

df = df.withColumn("doubled", double(col("amount")))

Pandas UDF (Vectorized — 10-100x faster!)

import pandas as pd
from pyspark.sql.functions import pandas_udf

# Scalar Pandas UDF: pd.Series → pd.Series
@pandas_udf("double")
def normalize(series: pd.Series) -> pd.Series:
    return (series - series.mean()) / series.std()

df = df.withColumn("normalized", normalize(col("amount")))

# Grouped Map UDF: pd.DataFrame → pd.DataFrame (per group)
def top_n_per_group(pdf: pd.DataFrame) -> pd.DataFrame:
    return pdf.nlargest(3, "sales")

result = df.groupBy("region").applyInPandas(top_n_per_group, schema=df.schema)

# Grouped Aggregate: pd.Series → scalar
@pandas_udf("double")
def gmean(series: pd.Series) -> float:
    return series.product() ** (1 / len(series))

df.groupBy("region").agg(gmean(col("amount"))).show()

When to Use UDFs

ScenarioApproach
Simple string/numeric transformUse built-in F.* functions
Complex Python library (e.g., sklearn)Pandas UDF
Per-row logic with simple conditionsF.when().otherwise()
Whatever-it-takes one-offRegular UDF (last resort)
Cryptographic / hashingUse F.sha2, F.md5 (built-in)

UDF Performance Comparison

# Same task, 3 approaches on 100M rows:

# ❌ Slowest: Python UDF
@udf("double")
def py_double(x): return x * 2
df.withColumn("d", py_double(col("amount")))    # ~120 seconds

# ✅ Fast: Pandas UDF
@pandas_udf("double")
def pd_double(s: pd.Series) -> pd.Series: return s * 2
df.withColumn("d", pd_double(col("amount")))    # ~15 seconds

# ✅✅ Fastest: Native expression
df.withColumn("d", col("amount") * 2)            # ~2 seconds

3. Caching / Persisting

from pyspark import StorageLevel
df.persist(StorageLevel.MEMORY_AND_DISK)
df.persist(StorageLevel.MEMORY_ONLY_SER)        # serialized (less memory)
df.persist(StorageLevel.DISK_ONLY)

# Unpersist when done!
df.unpersist()

# Rule: cache only if used 2+ times. Caching has cost.

4. Partition Sizing

# Default 200 partitions after shuffle — often wrong
spark.conf.set("spark.sql.shuffle.partitions", "400")    # bigger jobs

# Target: 100-200 MB per partition
# Calc: total_data_size / 128MB ≈ ideal partitions

#   1GB → 8-10 partitions

#   1TB → 8000-10000 partitions

5. Avoid Common Bottlenecks

❌ collect() on big data

# Brings ALL data to driver → OOM
df.collect()

# ✅ Sample / take
df.take(100)
df.sample(0.01).collect()

❌ toPandas() on big data

# Same problem as collect
df.toPandas()

# ✅ Filter first
df.limit(10000).toPandas()
# Or enable Arrow + sample
spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true")

❌ Repeated transformations

# Without cache, runs 3x
df = expensive_transform(...)
df.count()
df.show()
df.write.parquet(...)

# ✅ Cache once
df.cache(); df.count()  # materialize

❌ Python UDFs

# 10-100x slower
@udf("string")
def fn(x): return x.upper()

# ✅ Built-in
F.upper(col("x"))

❌ Skewed joins without handling

# Some partitions take 10x longer
big.join(other, "user_id")

# ✅ Filter nulls, use AQE, salt
big.filter(col("user_id").isNotNull())
   .join(broadcast(small), "user_id")

❌ Too many small files

# Slow listing, slow read
df.write.parquet(...)  # creates 200 small files

# ✅ Coalesce before write
df.coalesce(20).write.parquet(...)
# Or use Delta OPTIMIZE

10. Structured Streaming

Core Concept

Treat a stream as an infinite table. Apply DataFrame operations as if it were batch. Spark handles the micro-batch execution.

Read from Stream

# From Kafka
stream_df = (spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "broker:9092")
    .option("subscribe", "events_topic")
    .option("startingOffsets", "latest")              # or "earliest"
    .load()
)

# Kafka returns columns: key, value, topic, partition, offset, timestamp
# Value is binary → decode
parsed = stream_df.select(
    F.col("value").cast("string").alias("json_str"),
    F.col("timestamp")
)

# Parse JSON
from pyspark.sql.types import StructType, StringType, DoubleType
schema = StructType().add("user_id", StringType()).add("amount", DoubleType())
parsed = parsed.select(F.from_json("json_str", schema).alias("data"), "timestamp") \
               .select("data.*", "timestamp")

# From files (file source)
stream_df = (spark.readStream
    .schema(my_schema)
    .option("maxFilesPerTrigger", 1)
    .parquet("s3://bucket/incoming/")
)

# From Delta as stream
stream_df = spark.readStream.format("delta").load("s3://bucket/delta/")

Transformations (same as batch!)

aggregated = (parsed
    .filter(col("amount") > 0)
    .withWatermark("timestamp", "10 minutes")        # late data handling
    .groupBy(
        F.window("timestamp", "5 minutes"),
        col("user_id")
    )
    .agg(F.sum("amount").alias("total"))
)

Watermarking (handle late events)

# Tells Spark: drop events older than 10 minutes from the latest seen
.withWatermark("event_time", "10 minutes")

# Without watermark, aggregation state grows forever (memory leak)

Write Stream (Sinks)

# Console (debugging only)
query = (aggregated.writeStream
    .outputMode("append")               # append, update, complete
    .format("console")
    .option("truncate", "false")
    .start()
)

# To Kafka
query = (aggregated.writeStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "broker:9092")
    .option("topic", "output_topic")
    .option("checkpointLocation", "/tmp/checkpoint/")
    .start()
)

# To Parquet/Delta
query = (aggregated.writeStream
    .format("delta")
    .outputMode("append")
    .option("checkpointLocation", "/tmp/ckpt/")
    .start("s3://bucket/output/")
)

# Trigger types
.trigger(processingTime="30 seconds")    # micro-batch every 30s
.trigger(once=True)                       # one batch, then stop (Trigger.Once)
.trigger(availableNow=True)               # process all available now
.trigger(continuous="1 second")           # experimental low-latency mode

# foreachBatch — write to any sink (custom logic)
def process_batch(batch_df, batch_id):
    batch_df.write.format("jdbc").mode("append").save()

query = aggregated.writeStream.foreachBatch(process_batch).start()

# Wait + stop
query.awaitTermination()
query.stop()

11. Interview Q&A

Q1. RDD vs DataFrame vs Dataset?
RDD: low-level, no optimization, slowest. DataFrame: tabular API, Catalyst optimizer, recommended. Dataset: typed (Scala/Java only — not Python). For PySpark always use DataFrames.
Q2. Transformations vs Actions?
Transformations are lazy (select, filter, join, withColumn) — build DAG without executing. Actions trigger execution (show, count, collect, write, take).
Q3. What's the difference between cache() and persist()?
cache() is shortcut for persist(MEMORY_AND_DISK). persist() lets you specify storage level (MEMORY_ONLY, DISK_ONLY, MEMORY_AND_DISK_SER, etc.).
Q4. repartition vs coalesce?
Both change partition count. repartition(n) does a full shuffle, distributes evenly, can increase or decrease. coalesce(n) only decreases, avoids full shuffle by combining existing partitions. Use coalesce before write to reduce small files.
Q5. What is a shuffle? When does it happen?
Shuffle = redistributing data across executors over network. Triggered by groupBy, join, orderBy, repartition, distinct. It's expensive (network + disk I/O) — minimize shuffles for performance.
Q6. What is broadcast join?
Spark sends the entire small table to every executor, eliminating shuffle on the big table. Auto-triggered when one side < spark.sql.autoBroadcastJoinThreshold (default 10MB). Force with broadcast(df) hint.
Q7. What is data skew? How to handle it?
When one partition has way more data than others (often due to NULLs or hot keys), one task takes hours while others finish quickly. Solutions: filter null keys, enable AQE skew handling, use salting (add random suffix to spread keys), broadcast smaller side.
Q8. Why are Python UDFs slow?
Data must serialize between JVM and Python process for every row, breaking vectorization. Use built-in pyspark.sql.functions first; if not possible, use Pandas UDFs which are vectorized (~10-100x faster).
Q9. What is the Catalyst optimizer?
Spark SQL's query optimizer. Transforms logical plan via rules (predicate pushdown, column pruning, constant folding) then generates optimized physical plan.
Q10. What is AQE (Adaptive Query Execution)?
Re-optimizes the query during runtime based on actual statistics. Coalesces small post-shuffle partitions, switches join strategies (e.g., sort-merge → broadcast), splits skewed partitions. Default ON in Spark 3.x.
Q11. How do you choose number of partitions?
Target 100-200 MB per partition. Calc: total_size / 128MB. spark.sql.shuffle.partitions (default 200) often too low for big jobs. Each partition processed by one task, so partitions should be ≥ # cores for parallelism.
Q12. What is the difference between partitionBy and bucketBy?
partitionBy: writes to separate folders by column values; great for partition pruning. bucketBy: writes hash-partitioned files within folders; avoids shuffle for repeated joins on same key. Bucketing is less flexible (fixed at write).
Q13. Why Parquet over CSV?
Parquet: columnar (column pruning), compressed (5-10x smaller), schema-preserved, predicate pushdown. CSV: row-based, no schema, slow to parse, no compression by default. Always use Parquet/Delta in production.
Q14. What does collect() do? Why is it dangerous?
Brings all data from executors to driver as a list. Dangerous on big data → driver OOM. Use take(n), sample(), or write to storage instead.
Q15. What is checkpointing in Structured Streaming?
Stores stream progress (offsets), state (aggregations), and commits to durable storage. Enables exactly-once semantics, safe restart, and recovery from failures. Required for production streaming.
Q16. What is watermarking?
Threshold for late events: "ignore data older than X". Lets Spark drop old state (otherwise state grows forever). Trade-off: longer watermark = handles later data but uses more memory.
Q17. How does Delta MERGE work?
Performs UPSERT (insert + update + delete) atomically. Reads target + source, matches on condition, executes whenMatched/whenNotMatched clauses. Implements ACID via Delta's transaction log.
Q18. What is Z-Ordering?
Delta optimization that physically sorts data by multiple columns using Z-curve to maximize file skipping at read time. Use on columns frequently filtered together: OPTIMIZE table ZORDER BY (col1, col2).
Q19. How would you debug a slow Spark job?
Start with Spark UI: identify slowest stages, check task duration variance (skew), look at shuffle read/write sizes. Check explain() output for unexpected SortMerge joins, missing predicate pushdown, or broadcast that didn't happen.
Q20. Difference between repartition by hash and by range?
repartition(n, col): hash partition. Same value → same partition. repartitionByRange(n, col): range partition. Ordered, useful before orderBy to maintain order.
