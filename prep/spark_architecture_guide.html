⚡ Apache Spark Architecture

A visual, animated guide that explains Spark in plain English — from "what is it" to how the Driver, Cluster Manager, and Executors work together — plus interview questions at the end.

🧠 What is Apache Spark? (Layman's terms)

Imagine you have to count every word in a 10,000-page book.

One person doing it alone (a normal program on one computer) would take forever. Instead, you tear the book into 100 chunks and give them to 100 friends. Each friend counts their chunk at the same time, then you add up the totals. You finish ~100× faster.

That's exactly what Spark does — it splits huge data into pieces, processes them in parallel across many computers, and combines the results. Spark is the "manager" that organizes all those friends (computers) efficiently.

Apache Spark is an open-source distributed computing engine for processing very large amounts of data (gigabytes to petabytes) quickly, by spreading the work across a cluster of machines.

Why was it created? (Spark vs old Hadoop MapReduce)

Before Spark, people used Hadoop MapReduce. The big problem: after every step it wrote results to disk (hard drive), which is slow. Spark keeps data in memory (RAM) between steps, making it up to 100× faster for many workloads.

Analogy: MapReduce is like a chef who washes and puts away every knife and bowl after each ingredient. Spark keeps the tools on the counter (in memory) until the whole dish is done — far less wasted time.

Key things that make Spark special

  • In-memory computing — keeps data in RAM for speed.
  • Distributed — runs across many machines (a cluster).
  • Fault-tolerant — if a machine dies, Spark rebuilds the lost work automatically.
  • Lazy evaluation — it plans the whole recipe before cooking, so it can optimize.
  • Polyglot — write code in Python (PySpark), Scala, Java, or R.
One line to remember: Spark = a fast, in-memory engine that breaks big data into pieces and processes them in parallel across many computers, while automatically handling failures.

🏗️ Spark Architecture (Master–Worker model)

Spark uses a master–worker (also called manager–worker) architecture. There is one "brain" that plans and coordinates, and many "hands" that do the actual work.

Restaurant analogy: The Driver is the head chef who reads the order and creates a plan. The Cluster Manager is the restaurant manager who assigns kitchen stations (resources). The Executors are the line cooks who actually cook. Each Task is one small cooking step a cook performs.

The components, one by one

1. 🧠 Driver Program

The master process and the brain of the application. It runs your main() function and does the planning:

  • Creates the SparkSession / SparkContext (your entry point into Spark).
  • Converts your code into a DAG (a step-by-step execution plan).
  • Splits the work into Jobs → Stages → Tasks.
  • Schedules tasks onto executors and tracks their progress.
  • Collects final results.
If the Driver dies, the whole application dies. It's the single point of coordination.

2. 🗂️ Cluster Manager

The resource negotiator. It owns the cluster's CPUs and memory, and hands out resources (containers/executors) to applications. Spark can use different ones:

Cluster ManagerWhen used
StandaloneSpark's own simple built-in manager.
YARNMost common in Hadoop ecosystems.
KubernetesModern, container-based deployments.
MesosOlder general-purpose manager (now rare).

3. ⚙️ Executors

The worker processes (JVMs) that run on worker nodes. Each executor:

  • Runs the actual Tasks assigned by the Driver.
  • Has its own CPU cores and memory.
  • Stores cached/persisted data in memory or disk.
  • Reports results and status back to the Driver.

One executor can run multiple tasks at once — as many as it has cores.

4. 🔧 Tasks, Cores & Slots

A Task is the smallest unit of work — it processes one partition of data. The number of tasks an executor can run in parallel equals its number of cores (also called "slots").

5. 🌐 Worker Node

A physical/virtual machine in the cluster that hosts one or more executors.

Hierarchy: Cluster → Worker Nodes → Executors → Cores → Tasks → Partitions.

🔄 How a Spark Job Actually Runs

When you run Spark code, here is the journey from your code to results:

  1. You write code (transformations + an action).
  2. Driver builds a DAG — a logical graph of all operations.
  3. DAG Scheduler splits the DAG into Stages — boundaries happen at shuffles.
  4. Each Stage becomes a set of Tasks (one task per partition).
  5. Task Scheduler sends tasks to Executors via the Cluster Manager.
  6. Executors run tasks and return results to the Driver.

Job vs Stage vs Task:

  • Job = the whole assignment, triggered by one action (e.g. count()).
  • Stage = a group of steps that can run without reshuffling data. A new stage starts at every shuffle.
  • Task = one stage applied to one partition (the smallest unit).

📦 RDD, DataFrame & Dataset

These are the ways Spark represents your distributed data. All of them are split into partitions spread across executors.

🧱

RDD

Resilient Distributed Dataset. The original low-level API. Immutable, fault-tolerant collection. Full control, but no automatic optimization.

📊

DataFrame

Data organized into named columns (like a table). Optimized by Catalyst. The most-used API today.

🧩

Dataset

DataFrame + compile-time type safety (Scala/Java). Best of both worlds for typed languages.

RDD — the foundation (and why it's "Resilient")

  • Resilient → fault-tolerant; can rebuild lost partitions using lineage (the recipe of how it was built).
  • Distributed → data is partitioned across the cluster.
  • Dataset → a collection of records.
  • Immutable → never changed in place; transformations create new RDDs.
Lineage analogy: Spark doesn't back up your data like photocopies. Instead it remembers the recipe ("read file → filter → map"). If a partition is lost, it just re-runs the recipe for that piece. That's fault tolerance without expensive replication.

Partitions

A partition is a chunk of the data. It is the unit of parallelism — one task processes one partition. More partitions = more parallelism (up to the number of cores available).

# PySpark example
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("Demo").getOrCreate()

df = spark.read.csv("sales.csv", header=True)   # DataFrame
result = (df
    .filter(df.amount > 100)               # transformation (lazy)
    .groupBy("region")                    # transformation (lazy)
    .count())                              # transformation (lazy)

result.show()   # ACTION → now Spark actually runs everything

Catalyst & Tungsten (why DataFrames are fast)

  • Catalyst Optimizer — automatically rewrites your query plan to be more efficient (predicate pushdown, reordering, etc.).
  • Tungsten — optimizes memory and CPU usage with off-heap memory and code generation.

🐢 Lazy Evaluation, Transformations & Actions

Transformations vs Actions

Transformation (lazy)Action (triggers execution)
Builds the plan, runs nothing yetForces Spark to actually compute
map, filter, groupBy, join, selectcount, collect, show, save, take
Returns a new RDD/DataFrameReturns a value or writes output
Lazy evaluation analogy: It's like writing a shopping list (transformations) but not actually going to the store until someone says "go buy it" (an action). Because Spark sees the whole list first, it can plan the most efficient trip.

Narrow vs Wide Transformations

➡️

Narrow fast

Each input partition feeds exactly one output partition. No data moves between machines. e.g. map, filter.

🔀

Wide shuffle

Data must be redistributed across partitions/machines. Causes a shuffle (expensive). e.g. groupBy, join, reduceByKey.

🔀 What is a Shuffle?

A shuffle is when Spark moves data across the network so that related records (e.g. same key) end up on the same machine. It involves disk I/O, serialization, and network transfer — so it's the most expensive operation and a common cause of slow jobs.

Performance tip: Minimize shuffles. Filter early, use reduceByKey instead of groupByKey, and broadcast small tables in joins.

Persistence: cache() & persist()

If you reuse a DataFrame multiple times, cache() keeps it in memory so Spark doesn't recompute it from scratch every time.

df.cache()        # keep in memory (MEMORY_AND_DISK by default for DataFrames)
df.persist(StorageLevel.MEMORY_AND_DISK)
df.unpersist()     # free it when done

🧮 Executor Memory & Tuning Basics

Each executor's memory is divided into regions. Understanding this helps avoid the dreaded OutOfMemoryError.

RegionUsed for
Execution memoryShuffles, joins, sorts, aggregations.
Storage memoryCached/persisted data & broadcast variables.
User memoryYour data structures, UDF state.
Reserved memorySpark internal objects.

Execution and Storage share a "unified" pool and can borrow from each other.

Key configs you'll set

spark-submit \
  --num-executors 10 \        # how many executors
  --executor-cores 4 \        # cores (parallel tasks) per executor
  --executor-memory 8g \      # RAM per executor
  --driver-memory 4g \        # RAM for the driver
  my_app.py

Common performance levers

  • Partitioning — too few = underuse cluster; too many = scheduling overhead. spark.sql.shuffle.partitions defaults to 200.
  • Broadcast joins — send a small table to every executor to avoid shuffling a big one.
  • Avoid collect() on big data — it pulls everything to the driver and can crash it.
  • Cache wisely — only what you reuse.
  • Adaptive Query Execution (AQE) — Spark 3+ auto-tunes partitions and joins at runtime.
  • Data skew — uneven keys overload one task; fix with salting or AQE skew handling.
Rule of thumb: aim for partitions of ~128 MB and keep executors at 4–5 cores for good throughput without too much GC pressure.
