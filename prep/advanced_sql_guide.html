1. SQL Fundamentals (Quick Refresher)

Logical Query Processing Order

SQL is written in one order but executed in another. Understanding this is critical:

Written OrderExecuted Order
SELECT5. SELECT
FROM1. FROM / JOIN
WHERE2. WHERE
GROUP BY3. GROUP BY
HAVING4. HAVING
ORDER BY6. ORDER BY
LIMIT7. LIMIT
Why it matters: You cannot use a column alias defined in SELECT inside WHERE — because WHERE runs before SELECT. You CAN use it in ORDER BY.

WHERE vs HAVING

-- WHERE filters ROWS before aggregation
SELECT department, COUNT(*) AS emp_count
FROM employees
WHERE salary > 50000          -- filters individual rows
GROUP BY department;

-- HAVING filters GROUPS after aggregation
SELECT department, COUNT(*) AS emp_count
FROM employees
GROUP BY department
HAVING COUNT(*) > 10;         -- filters groups

NULL Handling (Common Bug Source!)

SQL uses three-valued logic (3VL): TRUE, FALSE, and UNKNOWN. Any comparison involving NULL — even NULL = NULL — returns UNKNOWN. In most engines this is displayed as NULL, and WHERE only keeps rows where the predicate is TRUE (UNKNOWN rows are filtered out).

NULL Comparison Truth Table

ExpressionResultWhy
NULL = NULLUNKNOWNNULL means "unknown value" — can't say if two unknowns are equal
5 = NULLUNKNOWNUnknown value could be anything, can't compare
NULL <> NULLUNKNOWNSame reason — can't say if two unknowns differ
5 <> NULLUNKNOWNCan't compare with unknown
NULL < 10UNKNOWNUnknown could be anything — could be < 10 or not
NULL > NULLUNKNOWNComparing two unknowns
NULL <= 5UNKNOWNSame logic
NULL >= 5UNKNOWNSame logic
NULL IS NULLTRUE ✅Special operator IS handles NULL explicitly
NULL IS NOT NULLFALSESame — handled by IS operator
NULL + 5NULLArithmetic with NULL propagates NULL
NULL || 'abc'NULL*String concat with NULL = NULL (except in Oracle, where NULL = '')

Three-Valued Logic — Boolean Operators

ABA AND BA OR B
TRUEUNKNOWNUNKNOWNTRUE
FALSEUNKNOWNFALSEUNKNOWN
UNKNOWNUNKNOWNUNKNOWNUNKNOWN
Key takeaway: FALSE AND UNKNOWN = FALSE (because false short-circuits). TRUE OR UNKNOWN = TRUE (because true short-circuits). All other UNKNOWN combinations stay UNKNOWN.
-- NULL is NOT equal to anything, including itself
SELECT NULL = NULL;          -- returns NULL/UNKNOWN (not TRUE!)
SELECT NULL = 0;             -- returns NULL/UNKNOWN
SELECT NULL != NULL;         -- returns NULL/UNKNOWN

-- Use IS NULL / IS NOT NULL — the only safe way
SELECT * FROM users WHERE email IS NULL;
SELECT * FROM users WHERE email IS NOT NULL;

-- IS DISTINCT FROM treats NULLs as equal (Postgres / Snowflake / ANSI)
SELECT NULL IS DISTINCT FROM NULL;          -- FALSE (they're "the same")
SELECT NULL IS NOT DISTINCT FROM NULL;      -- TRUE
SELECT 5 IS DISTINCT FROM NULL;             -- TRUE (they differ)
-- Useful in JOIN conditions: ON a.col IS NOT DISTINCT FROM b.col

-- COALESCE: returns first non-null value
SELECT COALESCE(phone, email, 'No contact') AS contact FROM users;

-- NULLIF: returns NULL if both are equal
SELECT NULLIF(salary, 0);    -- protects against divide-by-zero

-- IFNULL (MySQL) / ISNULL (SQL Server) / NVL (Oracle)
Gotcha: COUNT(*) includes NULLs. COUNT(column_name) excludes NULLs.

DISTINCT

SELECT DISTINCT department FROM employees;

-- DISTINCT applies to ALL columns in SELECT
SELECT DISTINCT department, job_title FROM employees;
-- Returns unique combinations of (department, job_title)

-- Counting distinct values
SELECT COUNT(DISTINCT customer_id) FROM orders;

LIMIT / OFFSET / FETCH

-- PostgreSQL / MySQL / Snowflake
SELECT * FROM products ORDER BY price DESC LIMIT 10 OFFSET 20;

-- ANSI standard / SQL Server / Oracle 12c+
SELECT * FROM products
ORDER BY price DESC
OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;

-- SQL Server (legacy)
SELECT TOP 10 * FROM products ORDER BY price DESC;

2. Joins — Combining Tables

Sample Tables (for examples)

-- employees
+----+-------+---------+
| id | name  | dept_id |
+----+-------+---------+
|  1 | Asha  |   10    |
|  2 | Ravi  |   20    |
|  3 | Priya |   NULL  |
+----+-------+---------+

-- departments
+----+-----------+
| id |   name    |
+----+-----------+
| 10 | Engineering|
| 20 | Sales     |
| 30 | HR        |
+----+-----------+

INNER JOIN — Only matching rows

SELECT e.name, d.name AS department
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;
-- Result: Asha-Engineering, Ravi-Sales
-- Priya is excluded (NULL dept_id)
-- HR is excluded (no employees)

LEFT JOIN — All rows from left + matching from right

SELECT e.name, d.name AS department
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;
-- Result: Asha-Engineering, Ravi-Sales, Priya-NULL
-- All employees kept, even Priya with no department

RIGHT JOIN — All from right + matching from left

SELECT e.name, d.name AS department
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.id;
-- Result: Asha-Engineering, Ravi-Sales, NULL-HR
-- All departments kept, including HR with no employees

FULL OUTER JOIN — All rows from both

SELECT e.name, d.name AS department
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;
-- Result: Asha-Engineering, Ravi-Sales, Priya-NULL, NULL-HR

CROSS JOIN — Cartesian product

SELECT e.name, d.name
FROM employees e
CROSS JOIN departments d;
-- Returns 3 × 3 = 9 rows (every combination)
-- Useful for generating date sequences, test data

SELF JOIN — Table joins to itself

-- Find employees and their managers
-- employees(id, name, manager_id)
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

ANTI JOIN (NOT EXISTS / LEFT JOIN ... IS NULL)

-- Find departments with NO employees
-- Method 1: NOT EXISTS (cleanest)
SELECT d.name
FROM departments d
WHERE NOT EXISTS (
  SELECT 1 FROM employees e WHERE e.dept_id = d.id
);

-- Method 2: LEFT JOIN ... IS NULL
SELECT d.name
FROM departments d
LEFT JOIN employees e ON e.dept_id = d.id
WHERE e.id IS NULL;

-- Method 3: NOT IN (BEWARE: breaks if right side has NULLs!)
SELECT name FROM departments
WHERE id NOT IN (SELECT dept_id FROM employees);  -- DANGER if dept_id can be NULL
Critical gotcha: NOT IN (subquery with NULL) returns 0 rows because x != NULL is unknown. Always use NOT EXISTS instead.

SEMI JOIN (EXISTS / IN)

-- Find departments that have at least one employee
SELECT d.name
FROM departments d
WHERE EXISTS (SELECT 1 FROM employees e WHERE e.dept_id = d.id);

-- Same with IN
SELECT name FROM departments
WHERE id IN (SELECT dept_id FROM employees WHERE dept_id IS NOT NULL);

Multiple Joins

SELECT o.order_id, c.name AS customer, p.name AS product, oi.quantity
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
INNER JOIN order_items oi ON oi.order_id = o.id
INNER JOIN products p ON oi.product_id = p.id
WHERE o.order_date >= '2026-01-01';

USING vs ON

-- ON: explicit, works with different column names
SELECT * FROM employees e JOIN departments d ON e.dept_id = d.id;

-- USING: shorthand when column names match
SELECT * FROM orders JOIN customers USING (customer_id);
-- Returns customer_id only ONCE in output

3. Window Functions Very Frequent

Window functions perform calculations across a "window" of rows related to the current row — without collapsing rows like GROUP BY does. This is the #1 differentiator in SQL interviews.

Syntax

function_name(args) OVER (
  PARTITION BY col1, col2     -- divides rows into groups
  ORDER BY col3                -- orders rows within each group
  ROWS BETWEEN ... AND ...     -- defines window frame (optional)
)

Ranking Functions

-- Sample data: sales(emp_id, region, amount)

SELECT emp_id, region, amount,
  ROW_NUMBER() OVER (PARTITION BY region ORDER BY amount DESC) AS row_num,
  RANK()       OVER (PARTITION BY region ORDER BY amount DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY region ORDER BY amount DESC) AS dense_rnk,
  NTILE(4)     OVER (PARTITION BY region ORDER BY amount DESC) AS quartile
FROM sales;

-- Differences:
-- ROW_NUMBER:  1, 2, 3, 4   (always unique)
-- RANK:        1, 2, 2, 4   (gaps after ties)
-- DENSE_RANK:  1, 2, 2, 3   (no gaps after ties)
-- NTILE(4):    distributes rows into 4 equal buckets
When to use which:
  • ROW_NUMBER — when you need unique ordering (deduplication, pagination)
  • RANK — Olympic-style ranking with gaps (2 silver → next is 4th)
  • DENSE_RANK — gap-less ranking (top N distinct salaries)
  • NTILE(n) — divide into n equal buckets (quartiles, deciles)

LAG and LEAD — Access previous/next row

-- Compare each day's sales with previous day
SELECT
  sale_date,
  amount,
  LAG(amount, 1) OVER (ORDER BY sale_date) AS prev_day_amount,
  LEAD(amount, 1) OVER (ORDER BY sale_date) AS next_day_amount,
  amount - LAG(amount, 1) OVER (ORDER BY sale_date) AS day_over_day_change
FROM daily_sales;

-- LAG/LEAD with default value
LAG(amount, 1, 0) OVER (ORDER BY sale_date)  -- returns 0 if no previous row

-- LAG with offset 7 for week-over-week
LAG(amount, 7) OVER (ORDER BY sale_date)

FIRST_VALUE, LAST_VALUE, NTH_VALUE

SELECT
  product, sale_date, amount,
  FIRST_VALUE(amount) OVER (
    PARTITION BY product ORDER BY sale_date
  ) AS first_sale,
  LAST_VALUE(amount) OVER (
    PARTITION BY product ORDER BY sale_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS last_sale
FROM sales;
LAST_VALUE trap: Without explicit frame UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING, default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — so LAST_VALUE returns current row, not the actual last!

Aggregate Functions as Windows

-- Running total
SELECT
  order_date,
  amount,
  SUM(amount) OVER (ORDER BY order_date) AS running_total,
  AVG(amount) OVER (ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling_7day_avg,
  COUNT(*) OVER (PARTITION BY customer_id) AS total_orders_by_customer
FROM orders;

Window Frame Clauses

-- ROWS: physical row count
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW           -- 3 rows total
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW   -- running total
ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING   -- remaining sum
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING  -- full partition

-- RANGE: value-based (used with ORDER BY date for date ranges)
RANGE BETWEEN INTERVAL '7' DAY PRECEDING AND CURRENT ROW

Real-World Examples

① Top N per group (Top 3 highest-paid per department)

SELECT *
FROM (
  SELECT name, department, salary,
         DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
  FROM employees
) ranked
WHERE rnk <= 3;

② Deduplication (keep latest record per customer)

SELECT *
FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY updated_at DESC) AS rn
  FROM customer_records
) t
WHERE rn = 1;

③ Find sessions (gap between events > 30 min)

WITH events_with_gap AS (
  SELECT user_id, event_time,
         event_time - LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time) AS gap
  FROM user_events
),
session_starts AS (
  SELECT *,
         CASE WHEN gap IS NULL OR gap > INTERVAL '30 minutes' THEN 1 ELSE 0 END AS is_new_session
  FROM events_with_gap
),
session_ids AS (
  SELECT *,
         SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY event_time) AS session_id
  FROM session_starts
)
SELECT user_id, session_id, MIN(event_time) AS session_start, MAX(event_time) AS session_end
FROM session_ids
GROUP BY user_id, session_id;

④ Percentage of total

SELECT product, amount,
       amount * 100.0 / SUM(amount) OVER () AS pct_of_total,
       amount * 100.0 / SUM(amount) OVER (PARTITION BY category) AS pct_of_category
FROM products;

⑤ Find gaps in a sequence (missing IDs)

SELECT id + 1 AS missing_start, next_id - 1 AS missing_end
FROM (
  SELECT id, LEAD(id) OVER (ORDER BY id) AS next_id
  FROM tickets
) t
WHERE next_id - id > 1;

4. CTEs & Subqueries

Common Table Expression (CTE) — Basic

WITH high_value_orders AS (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
  HAVING SUM(amount) > 10000
)
SELECT c.name, h.total
FROM customers c
JOIN high_value_orders h ON c.id = h.customer_id;

Multiple CTEs (chained)

WITH
monthly_sales AS (
  SELECT DATE_TRUNC('month', order_date) AS month, SUM(amount) AS total
  FROM orders
  GROUP BY DATE_TRUNC('month', order_date)
),
growth AS (
  SELECT month, total,
         LAG(total) OVER (ORDER BY month) AS prev_month,
         (total - LAG(total) OVER (ORDER BY month)) * 100.0
           / NULLIF(LAG(total) OVER (ORDER BY month), 0) AS growth_pct
  FROM monthly_sales
)
SELECT * FROM growth WHERE growth_pct > 10;

Recursive CTE — Hierarchies, Sequences

-- Generate dates from 2026-01-01 to 2026-12-31
WITH RECURSIVE dates AS (
  SELECT DATE '2026-01-01' AS dt
  UNION ALL
  SELECT dt + INTERVAL '1 day'
  FROM dates
  WHERE dt < DATE '2026-12-31'
)
SELECT * FROM dates;

-- Employee hierarchy (org chart)
WITH RECURSIVE org_chart AS (
  -- Anchor: top-level (CEO has no manager)
  SELECT id, name, manager_id, 1 AS level, name AS path
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive: find reports
  SELECT e.id, e.name, e.manager_id, oc.level + 1, oc.path || ' > ' || e.name
  FROM employees e
  JOIN org_chart oc ON e.manager_id = oc.id
)
SELECT * FROM org_chart ORDER BY level, name;

-- Bill of Materials (parts containing parts)
WITH RECURSIVE bom AS (
  SELECT part_id, component_id, quantity, 1 AS depth
  FROM parts_components
  WHERE part_id = 'PRODUCT_X'

  UNION ALL

  SELECT pc.part_id, pc.component_id, pc.quantity * b.quantity, b.depth + 1
  FROM parts_components pc
  JOIN bom b ON pc.part_id = b.component_id
)
SELECT component_id, SUM(quantity) AS total_qty
FROM bom
GROUP BY component_id;
Recursive CTE caution: Always include a stopping condition (e.g., depth limit, IS NULL check) or you'll get infinite recursion / errors.

Subquery Types

Scalar Subquery (returns single value)

SELECT name, salary,
       (SELECT AVG(salary) FROM employees) AS company_avg,
       salary - (SELECT AVG(salary) FROM employees) AS diff_from_avg
FROM employees;

Row Subquery

SELECT * FROM employees
WHERE (dept_id, salary) = (SELECT dept_id, MAX(salary) FROM employees WHERE dept_id = 10);

Table Subquery (in FROM)

SELECT dept, avg_sal
FROM (
  SELECT dept_id AS dept, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY dept_id
) AS dept_stats
WHERE avg_sal > 50000;

Correlated Subquery (runs per outer row)

-- Find employees earning above their department's average
SELECT e1.name, e1.salary, e1.dept_id
FROM employees e1
WHERE e1.salary > (
  SELECT AVG(e2.salary)
  FROM employees e2
  WHERE e2.dept_id = e1.dept_id    -- correlation with outer query
);
Correlated subquery vs window function: Window functions are usually faster than correlated subqueries. Above query rewritten: 
SELECT * FROM (
  SELECT name, salary, dept_id, AVG(salary) OVER (PARTITION BY dept_id) AS dept_avg
  FROM employees
) t WHERE salary > dept_avg;

CTE vs Subquery — When to Use What

Use CTE whenUse Subquery when
Logic is complex / multi-stepSimple, single-use logic
Need to reference result multiple timesUsed only once inline
Need recursionNo recursion needed
Improving readabilityInline brevity is fine

5. Advanced Aggregations

Standard Aggregates

SELECT
  COUNT(*),                    -- all rows incl. NULLs
  COUNT(column),               -- non-null values
  COUNT(DISTINCT column),
  SUM(amount),
  AVG(amount),
  MIN(amount), MAX(amount),
  STDDEV(amount),              -- standard deviation
  VARIANCE(amount),
  STRING_AGG(name, ', ')       -- Postgres / SQL Server (LISTAGG in Oracle, GROUP_CONCAT in MySQL)
FROM sales;

GROUP BY with multiple columns

SELECT region, product_category, SUM(amount) AS total
FROM sales
GROUP BY region, product_category
ORDER BY region, total DESC;

ROLLUP — Hierarchical subtotals

SELECT region, product, SUM(amount) AS total
FROM sales
GROUP BY ROLLUP (region, product);

-- Adds subtotals at:
--   1. (region, product) - regular groups
--   2. (region, NULL)    - subtotal per region
--   3. (NULL, NULL)      - grand total

CUBE — All possible combinations

SELECT region, product, SUM(amount)
FROM sales
GROUP BY CUBE (region, product);

-- Includes:
-- (region, product), (region, NULL), (NULL, product), (NULL, NULL)

GROUPING SETS — Custom grouping

SELECT region, product, SUM(amount)
FROM sales
GROUP BY GROUPING SETS (
  (region, product),    -- per region per product
  (region),             -- per region total
  ()                    -- grand total
);

-- Use GROUPING() to distinguish NULL vs subtotal NULL
SELECT
  COALESCE(region, 'ALL REGIONS') AS region,
  COALESCE(product, 'ALL PRODUCTS') AS product,
  SUM(amount),
  GROUPING(region) AS is_region_total,
  GROUPING(product) AS is_product_total
FROM sales
GROUP BY GROUPING SETS ((region, product), (region), (product), ());

PIVOT — Rows to Columns

-- Standard SQL (works everywhere using CASE)
SELECT
  product,
  SUM(CASE WHEN region = 'North' THEN amount ELSE 0 END) AS north,
  SUM(CASE WHEN region = 'South' THEN amount ELSE 0 END) AS south,
  SUM(CASE WHEN region = 'East'  THEN amount ELSE 0 END) AS east,
  SUM(CASE WHEN region = 'West'  THEN amount ELSE 0 END) AS west
FROM sales
GROUP BY product;

-- SQL Server / Oracle native PIVOT
SELECT * FROM sales
PIVOT (
  SUM(amount) FOR region IN ('North', 'South', 'East', 'West')
);

-- Snowflake
SELECT * FROM sales
PIVOT (SUM(amount) FOR region IN ('North', 'South', 'East', 'West'));

UNPIVOT — Columns to Rows

-- Standard SQL using UNION ALL
SELECT product, 'north' AS region, north AS amount FROM pivoted
UNION ALL
SELECT product, 'south' AS region, south AS amount FROM pivoted
UNION ALL
SELECT product, 'east'  AS region, east  AS amount FROM pivoted
UNION ALL
SELECT product, 'west'  AS region, west  AS amount FROM pivoted;

-- Snowflake / SQL Server native
SELECT * FROM pivoted
UNPIVOT (amount FOR region IN (north, south, east, west));

FILTER Clause (PostgreSQL / Snowflake / SQLite)

SELECT
  COUNT(*) AS total_orders,
  COUNT(*) FILTER (WHERE status = 'shipped') AS shipped_orders,
  COUNT(*) FILTER (WHERE status = 'cancelled') AS cancelled_orders,
  SUM(amount) FILTER (WHERE region = 'North') AS north_sales
FROM orders;

HAVING with Aggregates

-- Customers with > 5 orders and total spend > 10000
SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS total
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > 5 AND SUM(amount) > 10000;

6. Built-in Functions

String Functions

-- Length
LENGTH('hello')               -- 5 (Postgres/MySQL/Snowflake)
LEN('hello')                  -- SQL Server

-- Case
UPPER('hello'), LOWER('HELLO'), INITCAP('hello world')   -- 'Hello World'

-- Concatenation
CONCAT('Hello', ' ', 'World')
'Hello' || ' ' || 'World'     -- standard SQL

-- Substring
SUBSTRING('hello world' FROM 7 FOR 5)    -- 'world'
SUBSTR('hello', 1, 3)                    -- 'hel'
LEFT('hello', 3), RIGHT('hello', 3)      -- 'hel', 'llo'

-- Trim and pad
TRIM('   hello   ')           -- 'hello'
LTRIM('   hello'), RTRIM('hello   ')
LPAD('5', 3, '0')             -- '005'
RPAD('hi', 5, '!')            -- 'hi!!!'

-- Replace
REPLACE('hello world', 'world', 'SQL')   -- 'hello SQL'

-- Search
POSITION('llo' IN 'hello')    -- 3
INSTR('hello', 'llo')         -- 3 (Oracle/MySQL)
CHARINDEX('llo', 'hello')     -- SQL Server

-- Split
SPLIT_PART('a,b,c', ',', 2)   -- 'b' (Postgres/Snowflake)
STRING_SPLIT('a,b,c', ',')    -- SQL Server table-valued

-- Regex
REGEXP_REPLACE(email, '@.*', '@example.com')
REGEXP_LIKE(phone, '^\+91[0-9]{10}$')
REGEXP_SUBSTR('order-12345', '[0-9]+')   -- '12345'

Date/Time Functions

-- Current
CURRENT_DATE, CURRENT_TIMESTAMP, NOW()
SYSDATE                       -- Oracle
GETDATE()                     -- SQL Server

-- Extract parts
EXTRACT(YEAR FROM order_date)
EXTRACT(MONTH FROM order_date)
EXTRACT(DOW FROM order_date)  -- day of week
DATE_PART('hour', timestamp_col)

-- Truncate
DATE_TRUNC('month', order_date)    -- 2026-05-01 from 2026-05-22
DATE_TRUNC('week', order_date)
DATE_TRUNC('quarter', order_date)

-- Arithmetic
order_date + INTERVAL '7 days'
order_date - INTERVAL '1 month'
DATEADD(day, 7, order_date)        -- SQL Server
DATEDIFF(day, start_date, end_date)

-- Format
TO_CHAR(NOW(), 'YYYY-MM-DD HH24:MI:SS')
TO_DATE('2026-05-22', 'YYYY-MM-DD')

-- Age / interval
AGE(end_date, start_date)          -- Postgres
end_date - start_date              -- days between

-- Last day of month / day of week
LAST_DAY(order_date)
DAYNAME(order_date)

Numeric Functions

ROUND(3.14159, 2)             -- 3.14
CEIL(3.2), FLOOR(3.8)         -- 4, 3
ABS(-5)                        -- 5
MOD(10, 3) or 10 % 3          -- 1
POWER(2, 10)                   -- 1024
SQRT(16)                       -- 4
SIGN(-5)                       -- -1

-- Safe division
amount / NULLIF(quantity, 0)   -- returns NULL instead of error

Conditional Functions

-- CASE expression (most universal)
SELECT
  CASE
    WHEN age < 13 THEN 'child'
    WHEN age < 20 THEN 'teen'
    WHEN age < 60 THEN 'adult'
    ELSE 'senior'
  END AS age_group
FROM users;

-- Simple CASE
CASE status
  WHEN 'A' THEN 'Active'
  WHEN 'I' THEN 'Inactive'
  ELSE 'Unknown'
END

-- COALESCE / NULLIF
COALESCE(nickname, first_name, 'Anonymous')
NULLIF(amount, 0)             -- NULL if amount = 0

-- IIF (SQL Server) / IF (MySQL)
IIF(score > 50, 'pass', 'fail')
IF(score > 50, 'pass', 'fail')

-- GREATEST / LEAST
GREATEST(col1, col2, col3)    -- max of values per row
LEAST(col1, col2, col3)

Type Conversion

CAST('123' AS INTEGER)
CAST(price AS DECIMAL(10,2))
'123'::INTEGER                -- Postgres shorthand
CONVERT(INT, '123')           -- SQL Server
TO_NUMBER('123'), TO_DATE('2026-01-01', 'YYYY-MM-DD')

JSON Functions (PostgreSQL / Snowflake)

-- Sample: {"user": {"name": "Asha", "age": 30}, "items": ["a", "b"]}

-- Extract
SELECT data->'user'->>'name'           -- 'Asha' (Postgres, ->> returns text)
SELECT data:user.name::string          -- Snowflake
SELECT JSON_VALUE(data, '$.user.name') -- SQL Server / Oracle
SELECT data['user']['name']            -- Some dialects

-- Array element
SELECT data->'items'->>0               -- 'a' (Postgres)
SELECT data:items[0]                   -- Snowflake

-- Construct JSON
SELECT JSON_BUILD_OBJECT('name', name, 'age', age) FROM users;     -- Postgres
SELECT OBJECT_CONSTRUCT('name', name, 'age', age) FROM users;      -- Snowflake

-- Flatten array
SELECT id, t.value
FROM orders, LATERAL FLATTEN(input => items) t;    -- Snowflake

SELECT id, jsonb_array_elements(items) FROM orders;  -- Postgres

7. Performance & Optimization

Indexes

-- B-tree (default, for equality and range)
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- Composite (column order matters!)
CREATE INDEX idx_orders_cust_date ON orders(customer_id, order_date);
-- Useful for: WHERE customer_id = X AND order_date > Y
-- Also useful for: WHERE customer_id = X
-- NOT useful for: WHERE order_date > Y (alone)

-- Unique
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Partial / Filtered
CREATE INDEX idx_active_orders ON orders(customer_id) WHERE status = 'active';

-- Functional / Expression
CREATE INDEX idx_lower_email ON users(LOWER(email));

-- Covering / Include (Postgres / SQL Server)
CREATE INDEX idx_cover ON orders(customer_id) INCLUDE (amount, order_date);

-- Drop
DROP INDEX idx_orders_customer;
Index leftmost prefix rule: A composite index on (a, b, c) helps queries filtering by a, (a,b), or (a,b,c) — but NOT just b or c.

EXPLAIN PLAN

EXPLAIN SELECT * FROM orders WHERE customer_id = 100;
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 100;  -- runs query
EXPLAIN (FORMAT JSON, ANALYZE) SELECT ...;                     -- Postgres detailed

-- Look for:
-- Seq Scan         → full table scan (bad for large tables)
-- Index Scan       → using index (good)
-- Index Only Scan  → best (covered by index)
-- Nested Loop      → small joins OK; bad for large
-- Hash Join        → great for large unsorted joins
-- Merge Join       → great for pre-sorted data
-- Filter           → rows filtered AFTER scan (often missing index)

Common Performance Anti-Patterns

❌ Functions on indexed columns

-- BAD: index not used
WHERE YEAR(order_date) = 2026

-- GOOD: use range
WHERE order_date >= '2026-01-01'
  AND order_date < '2027-01-01'

❌ Leading wildcard in LIKE

-- BAD: index not used
WHERE name LIKE '%smith'

-- GOOD or use full-text/trigram index
WHERE name LIKE 'smith%'

❌ SELECT *

-- BAD: pulls unused columns
SELECT * FROM orders;

-- GOOD: only what you need
SELECT id, customer_id, amount FROM orders;

❌ OR conditions on different columns

-- BAD: hard to index
WHERE col1 = 'a' OR col2 = 'b'

-- GOOD: UNION ALL
SELECT ... WHERE col1 = 'a'
UNION ALL
SELECT ... WHERE col2 = 'b' AND col1 != 'a'

❌ Implicit type casting

-- BAD: customer_id is INT
WHERE customer_id = '100'   -- forces conversion

-- GOOD
WHERE customer_id = 100

❌ DISTINCT to hide bad joins

-- BAD: DISTINCT hiding cartesian
SELECT DISTINCT c.* FROM customers c, orders o
WHERE c.id = o.customer_id

-- GOOD: use EXISTS
SELECT * FROM customers c
WHERE EXISTS (SELECT 1 FROM orders WHERE customer_id = c.id)

Partitioning

-- Range partitioning by date (Postgres)
CREATE TABLE orders (
  id BIGSERIAL,
  customer_id INT,
  order_date DATE NOT NULL,
  amount DECIMAL
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2026_q1 PARTITION OF orders
  FOR VALUES FROM ('2026-01-01') TO ('2026-04-01');

CREATE TABLE orders_2026_q2 PARTITION OF orders
  FOR VALUES FROM ('2026-04-01') TO ('2026-07-01');

-- Benefits: queries with date filter only scan relevant partition (partition pruning)
-- Easier archival: DROP TABLE orders_2024_q1;

-- Hash partitioning (distribute evenly)
CREATE TABLE users (...) PARTITION BY HASH (user_id);

-- List partitioning (specific values)
CREATE TABLE sales PARTITION BY LIST (region);
CREATE TABLE sales_apac PARTITION OF sales FOR VALUES IN ('IN', 'SG', 'JP');

Materialized Views

-- Precomputed query result, stored physically
CREATE MATERIALIZED VIEW daily_sales_summary AS
SELECT order_date, region, SUM(amount) AS total
FROM orders
GROUP BY order_date, region;

-- Refresh (Postgres)
REFRESH MATERIALIZED VIEW daily_sales_summary;
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_sales_summary;  -- no read lock

-- Snowflake: auto-maintained
CREATE MATERIALIZED VIEW mv_sales AS SELECT ... FROM orders;

CREATE OR REPLACE PROCEDURE archive_old_orders(cutoff DATE)
LANGUAGE plpgsql AS $$

Function (returns value)

CREATE OR REPLACE FUNCTION get_customer_total(cust_id INT)
RETURNS DECIMAL AS $$
DECLARE
  total DECIMAL;
BEGIN
  SELECT COALESCE(SUM(amount), 0) INTO total
  FROM orders WHERE customer_id = cust_id;
  RETURN total;
END;
$$ LANGUAGE plpgsql;

SELECT get_customer_total(100);

Triggers

CREATE OR REPLACE FUNCTION audit_changes()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_log(table_name, action, row_id, changed_at, old_data, new_data)
  VALUES (
    TG_TABLE_NAME, TG_OP, NEW.id, NOW(),
    row_to_json(OLD), row_to_json(NEW)
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_orders_audit
AFTER INSERT OR UPDATE OR DELETE ON orders
FOR EACH ROW EXECUTE FUNCTION audit_changes();
Triggers caveat: They're invisible logic — easy to forget. Prefer application-level logic for business rules.

9. Data Engineering SQL Patterns Critical for DE Interviews

MERGE / UPSERT (Insert-or-Update)

-- ANSI SQL / Snowflake / SQL Server
MERGE INTO target t
USING staging s ON t.id = s.id
WHEN MATCHED THEN
  UPDATE SET t.name = s.name, t.updated_at = NOW()
WHEN NOT MATCHED THEN
  INSERT (id, name, created_at) VALUES (s.id, s.name, NOW());

-- PostgreSQL (ON CONFLICT)
INSERT INTO customers (id, name, email)
VALUES (100, 'Asha', 'asha@example.com')
ON CONFLICT (id) DO UPDATE
SET name = EXCLUDED.name, updated_at = NOW();

-- MySQL
INSERT INTO customers (id, name) VALUES (100, 'Asha')
ON DUPLICATE KEY UPDATE name = VALUES(name);

SCD Type 1 — Overwrite (no history)

-- Just update the target row
MERGE INTO dim_customer t
USING stg_customer s ON t.customer_id = s.customer_id
WHEN MATCHED AND (t.name != s.name OR t.email != s.email) THEN
  UPDATE SET t.name = s.name, t.email = s.email, t.updated_at = NOW()
WHEN NOT MATCHED THEN
  INSERT (customer_id, name, email) VALUES (s.customer_id, s.name, s.email);

SCD Type 2 — Full history with version flag

-- dim_customer has: customer_id, name, email, valid_from, valid_to, is_current

-- Step 1: close out changed records
UPDATE dim_customer t
SET valid_to = CURRENT_DATE - INTERVAL '1 day', is_current = FALSE
FROM stg_customer s
WHERE t.customer_id = s.customer_id
  AND t.is_current = TRUE
  AND (t.name != s.name OR t.email != s.email);

-- Step 2: insert new versions for changed + brand-new records
INSERT INTO dim_customer (customer_id, name, email, valid_from, valid_to, is_current)
SELECT s.customer_id, s.name, s.email, CURRENT_DATE, NULL, TRUE
FROM stg_customer s
LEFT JOIN dim_customer t
  ON t.customer_id = s.customer_id AND t.is_current = TRUE
WHERE t.customer_id IS NULL
   OR (t.name != s.name OR t.email != s.email);

SCD Type 3 — Limited history (previous value column)

-- dim_customer: customer_id, name, current_email, previous_email, change_date
UPDATE dim_customer t
SET previous_email = t.current_email,
    current_email = s.email,
    change_date = CURRENT_DATE
FROM stg_customer s
WHERE t.customer_id = s.customer_id
  AND t.current_email != s.email;

Deduplication (keep latest)

DELETE FROM events e
USING (
  SELECT id, ROW_NUMBER() OVER (PARTITION BY event_key ORDER BY received_at DESC) AS rn
  FROM events
) ranked
WHERE e.id = ranked.id AND ranked.rn > 1;

-- Or in a query (keep only latest)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY event_key ORDER BY received_at DESC) AS rn
  FROM events
) t WHERE rn = 1;

Incremental Load Pattern

-- Watermark-based incremental
INSERT INTO target_table
SELECT * FROM source_table
WHERE updated_at > (SELECT MAX(updated_at) FROM target_table);

-- With staging + merge for idempotency
CREATE TEMPORARY TABLE stg AS
SELECT * FROM source_table
WHERE updated_at > '{{ last_run_time }}';

MERGE INTO target USING stg ON target.id = stg.id
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT ...;

CDC (Change Data Capture) Replay

-- From a changelog (id, op, after_data, op_ts)
WITH latest AS (
  SELECT id, op, after_data, op_ts,
         ROW_NUMBER() OVER (PARTITION BY id ORDER BY op_ts DESC) AS rn
  FROM changes
)
MERGE INTO target t
USING (SELECT * FROM latest WHERE rn = 1) c ON t.id = c.id
WHEN MATCHED AND c.op = 'U' THEN UPDATE SET t.data = c.after_data
WHEN MATCHED AND c.op = 'D' THEN DELETE
WHEN NOT MATCHED AND c.op IN ('I', 'U') THEN INSERT (id, data) VALUES (c.id, c.after_data);

Data Quality Checks

-- Null check
SELECT COUNT(*) AS null_count FROM customers WHERE email IS NULL;

-- Duplicate check
SELECT customer_id, COUNT(*) AS cnt FROM customers
GROUP BY customer_id HAVING COUNT(*) > 1;

-- Referential integrity
SELECT o.customer_id FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id
WHERE c.id IS NULL;

-- Range check
SELECT * FROM orders WHERE amount < 0 OR amount > 1000000;

-- Freshness
SELECT MAX(updated_at) AS last_update FROM orders;
-- Alert if < NOW() - INTERVAL '1 hour'

-- Row count anomaly (compared to yesterday)
WITH daily AS (
  SELECT DATE(created_at) AS d, COUNT(*) AS c FROM orders GROUP BY DATE(created_at)
)
SELECT d, c, AVG(c) OVER (ORDER BY d ROWS BETWEEN 7 PRECEDING AND 1 PRECEDING) AS avg_7d
FROM daily;

Date Spine (gap filling)

-- Generate continuous dates and LEFT JOIN actuals
WITH RECURSIVE dates AS (
  SELECT DATE '2026-01-01' AS dt
  UNION ALL
  SELECT dt + INTERVAL '1 day' FROM dates WHERE dt < DATE '2026-12-31'
)
SELECT d.dt, COALESCE(SUM(o.amount), 0) AS daily_total
FROM dates d
LEFT JOIN orders o ON DATE(o.order_date) = d.dt
GROUP BY d.dt
ORDER BY d.dt;

10. Top Interview Patterns Asked 80% of Time

Pattern 1: Nth Highest Salary

-- Nth highest salary using DENSE_RANK
WITH ranked AS (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
)
SELECT DISTINCT salary FROM ranked WHERE rnk = 3;

-- Alternative (subquery)
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);  -- 2nd highest

-- LIMIT + OFFSET (Postgres/MySQL)
SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 2;

Pattern 2: Top N per Group

-- Top 3 highest-paid per department
SELECT * FROM (
  SELECT name, dept, salary,
         DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS r
  FROM employees
) t WHERE r <= 3;

Pattern 3: Find Duplicates

SELECT email, COUNT(*) AS dup_count
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- Find which rows are dups (keep one)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY email ORDER BY created_at) AS rn
  FROM users
) t WHERE rn > 1;

Pattern 4: Consecutive Logins / Streaks

-- Find users with 3+ consecutive day logins
WITH numbered AS (
  SELECT user_id, login_date,
         login_date - INTERVAL '1 day' * ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date) AS grp
  FROM logins
),
streaks AS (
  SELECT user_id, grp, COUNT(*) AS streak_length,
         MIN(login_date) AS streak_start, MAX(login_date) AS streak_end
  FROM numbered
  GROUP BY user_id, grp
)
SELECT * FROM streaks WHERE streak_length >= 3;

Pattern 5: Median

-- Using PERCENTILE_CONT
SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees;

-- Per group
SELECT dept, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median_sal
FROM employees GROUP BY dept;

-- Manual (no PERCENTILE_CONT)
WITH ordered AS (
  SELECT salary, ROW_NUMBER() OVER (ORDER BY salary) AS rn,
         COUNT(*) OVER () AS cnt
  FROM employees
)
SELECT AVG(salary) AS median
FROM ordered
WHERE rn IN ((cnt + 1) / 2, (cnt + 2) / 2);

Pattern 6: Pivot Without PIVOT

SELECT
  product,
  SUM(CASE WHEN quarter = 'Q1' THEN amount END) AS Q1,
  SUM(CASE WHEN quarter = 'Q2' THEN amount END) AS Q2,
  SUM(CASE WHEN quarter = 'Q3' THEN amount END) AS Q3,
  SUM(CASE WHEN quarter = 'Q4' THEN amount END) AS Q4
FROM sales
GROUP BY product;

Pattern 7: Running Total / Cumulative

SELECT order_date, amount,
       SUM(amount) OVER (ORDER BY order_date) AS running_total,
       SUM(amount) OVER (PARTITION BY customer_id ORDER BY order_date) AS cust_running
FROM orders;

Pattern 8: Month-over-Month Growth

WITH monthly AS (
  SELECT DATE_TRUNC('month', order_date) AS mth, SUM(amount) AS total
  FROM orders GROUP BY 1
)
SELECT mth, total,
       LAG(total) OVER (ORDER BY mth) AS prev_total,
       (total - LAG(total) OVER (ORDER BY mth)) * 100.0
         / NULLIF(LAG(total) OVER (ORDER BY mth), 0) AS growth_pct
FROM monthly;

Pattern 9: Customers Who Bought X AND Y

SELECT customer_id
FROM orders
WHERE product IN ('A', 'B')
GROUP BY customer_id
HAVING COUNT(DISTINCT product) = 2;

Pattern 10: Find Users Active in N Consecutive Months

WITH user_months AS (
  SELECT DISTINCT user_id, DATE_TRUNC('month', activity_date) AS mth FROM activity
),
gaps AS (
  SELECT user_id, mth,
         mth - INTERVAL '1 month' * ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY mth) AS grp
  FROM user_months
)
SELECT user_id, COUNT(*) AS consec_months, MIN(mth), MAX(mth)
FROM gaps GROUP BY user_id, grp HAVING COUNT(*) >= 3;

Pattern 11: Department with Highest Average Salary

SELECT dept FROM (
  SELECT dept, AVG(salary) AS avg_sal,
         RANK() OVER (ORDER BY AVG(salary) DESC) AS r
  FROM employees GROUP BY dept
) t WHERE r = 1;

Pattern 12: Self-Join (Employee-Manager)

SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Employees earning more than their manager
SELECT e.name FROM employees e
JOIN employees m ON e.manager_id = m.id
WHERE e.salary > m.salary;

Pattern 13: Find Missing Numbers / Gaps

SELECT id + 1 AS gap_start, next_id - 1 AS gap_end
FROM (SELECT id, LEAD(id) OVER (ORDER BY id) AS next_id FROM tickets) t
WHERE next_id - id > 1;

Pattern 14: Last Activity per Customer

SELECT customer_id, last_order_date, total_orders
FROM (
  SELECT customer_id,
         MAX(order_date) OVER (PARTITION BY customer_id) AS last_order_date,
         COUNT(*) OVER (PARTITION BY customer_id) AS total_orders,
         ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
) t WHERE rn = 1;

Pattern 15: Cohort Retention

WITH cohort AS (
  SELECT user_id, DATE_TRUNC('month', MIN(activity_date)) AS cohort_month
  FROM activity GROUP BY user_id
),
activity_months AS (
  SELECT DISTINCT user_id, DATE_TRUNC('month', activity_date) AS active_month FROM activity
)
SELECT c.cohort_month,
       EXTRACT(MONTH FROM AGE(am.active_month, c.cohort_month)) AS months_since_signup,
       COUNT(DISTINCT am.user_id) AS active_users
FROM cohort c
JOIN activity_months am ON c.user_id = am.user_id
GROUP BY c.cohort_month, months_since_signup
ORDER BY c.cohort_month, months_since_signup;

💬 Common Q&A

Q1. What is the difference between RANK, DENSE_RANK, and ROW_NUMBER?
ROW_NUMBER assigns a unique sequential number. RANK assigns same rank to ties but skips next ranks (1, 2, 2, 4). DENSE_RANK assigns same rank to ties without gaps (1, 2, 2, 3).
Q2. What is the difference between WHERE and HAVING?
WHERE filters rows BEFORE aggregation. HAVING filters AFTER aggregation. WHERE cannot use aggregate functions; HAVING can.
Q3. What is a CTE? Why use it over a subquery?
A CTE (Common Table Expression) is a named temporary result set defined with WITH. Use CTEs for readability, when reusing the same logic multiple times in a query, or for recursive queries (which subqueries can't do).
Q4. What is the difference between UNION and UNION ALL?
UNION removes duplicates (expensive sort). UNION ALL keeps all rows including duplicates (faster). Use UNION ALL whenever duplicates aren't a problem.
Q5. What is the difference between DELETE, TRUNCATE, and DROP?
DELETE removes rows (row-by-row, can use WHERE, can rollback). TRUNCATE removes all rows fast (DDL, no WHERE, may not rollback). DROP removes the entire table including structure.
Q6. What is an index? When NOT to use one?
An index is a data structure (typically B-tree) that speeds up lookups at the cost of slower writes and more storage. Avoid indexes on small tables, heavily-written tables, or columns with low cardinality (e.g., boolean).
Q7. What's the difference between clustered and non-clustered indexes?
Clustered: data physically sorted by index key (one per table; PRIMARY KEY in many DBs). Non-clustered: separate structure pointing to row locations (multiple per table allowed).
Q8. Explain ACID properties.
Atomicity (all-or-nothing), Consistency (valid state transitions), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes).
Q9. What is normalization? What are 1NF, 2NF, 3NF?
Normalization reduces redundancy. 1NF: atomic values, no repeating groups. 2NF: 1NF + no partial dependency on composite key. 3NF: 2NF + no transitive dependency.
Q10. Star schema vs Snowflake schema?
Star: central fact table with denormalized dimensions (faster reads). Snowflake: dimensions are normalized into sub-dimensions (less redundancy but more joins).
Q11. What is SCD Type 2?
Slowly Changing Dimension Type 2 tracks full history by inserting new rows for changes and using valid_from/valid_to/is_current columns to mark versions.
Q12. How would you handle a slow-running query?
Use EXPLAIN ANALYZE; check for full scans; add or fix indexes; rewrite to avoid functions on indexed columns; use covering indexes; partition large tables; check stats are fresh; consider materialized views for expensive aggregations.
Q13. Difference between NOT IN and NOT EXISTS?
NOT IN returns 0 rows if the subquery has any NULL (due to 3-valued logic). NOT EXISTS handles NULLs correctly. Always prefer NOT EXISTS for anti-joins.
Q14. What is a materialized view?
A view whose result is physically stored (cached). Faster reads but needs periodic refresh. Useful for expensive aggregations queried frequently.
Q15. Difference between OLTP and OLAP?
OLTP (Online Transaction Processing): many small writes, normalized, low latency (e.g., Postgres for app DB). OLAP (Online Analytical Processing): bulk reads, denormalized/columnar, complex aggregations (e.g., Snowflake, Redshift).
